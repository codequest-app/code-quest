## Why

Two issues with the current `raw_entries` table conspire to motivate this change:

**1. Wrong name.** Everything else in the codebase talks about **events** — `RawEventStore` class, `raw:event` socket event, `stream_event` payload type, `subscribeRawEvents` client method, `RawEventPanel` component. The DB table is the lone outlier named `raw_entries`.

**2. Mixed concerns.** Streaming `content_block_delta` rows account for ~82% (117k / 143k) of all rows today. Chat replay doesn't need them (CLI emits authoritative `type: "assistant"` snapshots; client's `wasStreamedViaDelta` dedup already makes replay take the snapshot path when deltas are absent). Deltas exist solely for developer-only `RawEventPanel` debugging. Keeping them in the same table bloats backups, index sizes, and replay queries.

Since we're already doing DB migration for the rename, it costs nothing extra to also split deltas into a dedicated `raw_deltas` table at the same time. One migration, one PR, consistent end state.

## What Changes

### Schema (one migration, two operations)

- **Rename** `raw_entries` → `raw_events` (same columns, same indexes).
- **Create** `raw_deltas` table: `raw_events` columns + `parent_id` (FK-like reference to `raw_events.id`).
- **Drop** all `content_block_delta` rows that used to live in `raw_entries` — reclaiming ~30 MB and starting the new tables clean. Rows are unused (chat replay ignores them when absent; RawEventPanel is opt-in going forward).

### Env

- **New** `RAW_EVENTS_PERSIST_DELTAS` (default `false`). When `false`, delta lines are classified and dropped (not written to either table). When `true`, they go to `raw_deltas` with the current turn's `parent_id`.

### Code

- Schema/type rename: `rawEntries` → `rawEvents`, `RawEntry` → `RawEvent`, `RAW_ENTRY_COLUMNS` → `RAW_EVENT_COLUMNS`, `RawEntryColumnName` → `RawEventColumnName`.
- New `rawDeltas` schema + `RawDeltaStore` (parallel to `RawEventStore`).
- `RawEventStore.append()` returns the inserted id (`Promise<string>`). Needed so `RawRecorder` can stamp each delta's `parent_id` with the user stdin row's id.
- **`RawRecorder.wire(channel)` closure** tracks `currentTurnRootId` alongside existing `seqCounter` — no new `Channel` field.
- Classification helper `isDelta(raw: string)` deciding routing.
- `session:raw_events` RPC merges `raw_events` + `raw_deltas` by `seq` when `RAW_EVENTS_PERSIST_DELTAS` is on.

### Migration

- Drizzle migration 0017 (sqlite + mysql): rename `raw_entries` → `raw_events`, drop delta rows, create `raw_deltas`.
- Written by hand (drizzle-kit treats rename as drop+create, which would lose data). Snapshot files regenerated by `db:generate` after rename so drizzle's state machine tracks it.

## Capabilities

### New Capabilities

- `raw-event-recording`: behaviour contract for server-side persistence of CLI raw protocol events, including terminal-vs-delta separation and turn correlation via `parent_id`.

### Modified Capabilities

(none — no existing spec covers this area)

## Impact

- `packages/server/drizzle/{sqlite,mysql}/0017_*.sql` — hand-written migration.
- `packages/server/src/db/schema-{sqlite,mysql,columns}.ts` — rename existing table + add new `rawDeltas`.
- `packages/server/src/services/drizzle-raw-event-store.ts` — point at `rawEvents` table, `append` returns id.
- `packages/server/src/services/drizzle-raw-delta-store.ts` (new).
- `packages/server/src/services/composite-raw-event-store.ts` — propagate `append` id return.
- `packages/server/src/socket/raw-recorder.ts` — classification + routing + closure-local `currentTurnRootId`.
- `packages/server/src/config.ts` — add `persistDeltas: boolean` under `config.rawEvents`.
- `packages/server/src/container.ts` — wire `RawDeltaStore`.
- `packages/server/src/socket/handlers/session/query.ts` — conditional UNION for RawEventPanel.
- `packages/summoner/src/types.ts` — `RawEntry` → `RawEvent`.
- Tests: classifier unit, store unit, recorder routing, RPC integration, migration smoke.
- `.env` / `.env.example` — uncomment the `RAW_EVENTS_PERSIST_DELTAS=false` placeholder.
- **No client change. No protocol change.** Chat replay is unaffected.
