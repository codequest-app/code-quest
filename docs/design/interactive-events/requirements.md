## 核心概念

### 問題定義

在戰鬥過程中，Claude Code 可能觸發需要用戶互動的事件：

| 事件類型 | 觸發時機 | 原始行為 | 頻率 |
|---------|---------|---------|------|
| **Plan Mode** | EnterPlanMode tool 被調用 | 等待用戶批准計劃 | 中等複雜度任務 |
| **AskUserQuestion** | 需要用戶決策 | 顯示選項，等待選擇 | 不確定時 |
| **錯誤/警告** | 執行失敗 | 顯示錯誤訊息 | 發生錯誤時 |
| **權限請求** | 需要額外權限 | 請求用戶授權 | 敏感操作時 |

**挑戰**:
1. ❌ 這些事件會中斷戰鬥流程
2. ❌ 原始 UI 不符合 RPG 風格
3. ❌ 用戶可能不理解為何戰鬥暫停
4. ❌ 缺乏遊戲化元素

**設計目標**:
1. ✅ 保持原有功能正常運作
2. ✅ 用 RPG 元素包裝互動事件
3. ✅ 不破壞沉浸感
4. ✅ 提供清晰的視覺反饋

## 設計方案對比

### 方案 A: 戰鬥暫停（保守）

**概念**: 將互動事件視為「戰鬥暫停」

```
戰鬥進行中
    ↓
觸發互動事件（Plan Mode / Ask Question）
    ↓
┌─────────────────────────┐
│ ⏸️ 戰鬥暫停            │
│                         │
│ [顯示暫停原因]          │
│ [顯示互動內容]          │
│ [等待用戶回應]          │
└─────────────────────────┘
    ↓
用戶完成互動
    ↓
▶️ 戰鬥繼續
```

**優點**:
- ✅ 簡單直接
- ✅ 不會混淆用戶
- ✅ 實作容易
- ✅ 可快速推出（Phase 2.5）

**缺點**:
- ❌ 破壞沉浸感
- ❌ 不夠 RPG 化
- ❌ 沒有利用遊戲化元素
- ❌ 缺乏互動趣味性

**適用場景**:
- MVP 版本
- 快速實現基礎功能
- 作為過渡方案

---

### 方案 B: RPG 化事件（推薦）⭐

**概念**: 將互動事件轉化為戰鬥中的特殊事件

#### 1. Plan Mode → 「戰術規劃」

**遊戲化包裝**:
```
敵人：中等的架構挑戰 Lv.8
     ↓
Claude 進入 Plan Mode
     ↓
┌─────────────────────────────────────┐
│ 🧙 魔法師進入思考狀態...             │
│                                     │
│ 敵人太過強大，需要制定戰術！         │
│                                     │
│ ┌─────────────────────────────┐     │
│ │ 📋 戰術規劃                  │     │
│ │                             │     │
│ │ [顯示 Plan Mode 內容]        │     │
│ │                             │     │
│ │ 建議的戰術:                  │     │
│ │ 1. 分析敵人弱點              │     │
│ │ 2. 準備強力技能              │     │
│ │ 3. 召喚夥伴協助              │     │
│ │                             │     │
│ │ [✅ 批准戰術] [❌ 重新規劃]  │     │
│ └─────────────────────────────┘     │
└─────────────────────────────────────┘
```

**設計元素**:
- 🧙 圖標: 魔法師思考、📋 戰術書
- 🎨 視覺: 敵人進入「警戒」狀態（HP 條變黃色）
- 🎵 音效: 思考音效、翻書音效
- ✨ 動畫: 魔法師周圍浮現符文、戰術圖展開

---

#### 2. AskUserQuestion → 「敵人發問攻擊」

**遊戲化包裝**:
```
戰鬥進行中
     ↓
敵人使用特殊技能："困惑之問"
     ↓
┌─────────────────────────────────────┐
│ ❓ 敵人的發問攻擊！                  │
│                                     │
│ 敵人發動「困惑之問」，你必須正確     │
│ 回答才能繼續戰鬥！                   │
│                                     │
│ ┌─────────────────────────────┐     │
│ │ 問題:                        │     │
│ │ "應該使用哪種資料庫？"       │     │
│ │                             │     │
│ │ [A] MySQL                   │     │
│ │ [B] PostgreSQL              │     │
│ │ [C] MongoDB                 │     │
│ │ [D] SQLite                  │     │
│ └─────────────────────────────┘     │
│                                     │
│ ⚠️ 警告: 錯誤選擇可能造成代價！     │
└─────────────────────────────────────┘
```

**設計元素**:
- ❓ 圖標: 問號、對話泡泡
- 🎨 視覺: 敵人蓄力狀態（發光效果）
- 🎵 音效: 發問音效、選擇音效
- ✨ 動畫: 問題浮現、選項高亮

**選擇後果**:
- ✅ 正確回答 → 發現敵人弱點，額外傷害
- ❌ 錯誤回答 → 受到反噬傷害，降低士氣

---

#### 3. 錯誤/警告 → 「技能反噬」

**遊戲化包裝**:
```
施放技能: "代碼重構術"
     ↓
執行失敗（語法錯誤）
     ↓
┌─────────────────────────────────────┐
│ ⚠️ 技能反噬！                       │
│                                     │
│ 施法失敗，魔法反彈！                 │
│                                     │
│ ┌─────────────────────────────┐     │
│ │ 錯誤原因:                    │     │
│ │ Syntax Error at line 42     │     │
│ │                             │     │
│ │ 你受到了 15 點反噬傷害！      │     │
│ │ HP: 80 → 65                 │     │
│ └─────────────────────────────┘     │
│                                     │
│ [🔄 重試] [🛡️ 防禦] [❌ 取消]    │
└─────────────────────────────────────┘
```

**設計元素**:
- ⚠️ 圖標: 警告、閃電
- 🎨 視覺: HP 條閃爍紅色、震動效果
- 🎵 音效: 爆炸音效、傷害音效
- ✨ 動畫: 紅色波紋擴散

**機制**:
- 反噬傷害 = 錯誤嚴重度 × 5
- 連續錯誤會增加傷害倍率
- 可選擇重試、防禦或取消

```
│ │                             │     │

│                                     │
│ [✅ 授予力量] [❌ 拒絕]             │
└─────────────────────────────────────┘
```

**設計元素**:
- 🔐 圖標: 鎖、鑰匙
- 🎨 視覺: 金色光環、權限標誌
- 🎵 音效: 解鎖音效、神聖音效
- ✨ 動畫: 光芒閃耀

### 方案 C: 戰前處理（折衷）

**概念**: 某些互動事件在戰鬥開始前處理

```
用戶發起任務: "重構認證系統"
    ↓
系統判斷: 複雜度高，可能需要 Plan Mode
    ↓
┌─────────────────────────────────────┐
│ 📋 戰前準備                          │
│                                     │
│ 在開始戰鬥之前，制定戰術計劃...      │
│                                     │
│ [顯示 Plan Mode 內容]                │
│                                     │
│ [✅ 開始戰鬥] [❌ 取消]             │
└─────────────────────────────────────┘
    ↓
用戶批准計劃
    ↓
⚔️ 戰鬥開始（不會被 Plan Mode 中斷）
```

**優點**:
- ✅ 不會中斷戰鬥流程
- ✅ 用戶可提前規劃
- ✅ 減少戰鬥中的暫停

**缺點**:
- ❌ 並非所有事件都能預先處理
- ❌ AskUserQuestion 仍需要戰鬥中處理
- ❌ 可能增加戰鬥前的等待時間

**適用場景**:
- Plan Mode（戰鬥前預測）
- 初始權限請求

## 推薦方案

### 最佳實踐：混合方案 ⭐

**策略**: 根據事件類型和時機選擇最合適的處理方式

#### A. 互動事件（需要用戶決策）

| 事件類型 | 處理方式 | 遊戲化呈現 | 優先級 |
|---------|---------|-----------|--------|
| Plan Mode (戰鬥前) | 方案 C - 戰前準備 | 📋 制定戰術 | P1 |
| Plan Mode (戰鬥中) | 方案 B - RPG 化事件 | 🧙 戰術規劃 | P2 |
| AskUserQuestion | 方案 B - RPG 化事件 | ❓ 敵人發問攻擊 | P1 |
| 錯誤/警告 | 方案 B - RPG 化事件 | ⚠️ 技能反噬 | P1 |
| 權限請求 | 方案 B - RPG 化事件 | 🔐 力量借用 | P2 |

**決策規則**:

1. **Plan Mode**:
   - IF 戰鬥前可預測 → 使用方案 C（戰前處理）
   - IF 戰鬥中觸發 → 使用方案 B（RPG 化事件）

2. **AskUserQuestion**:
   - 始終使用方案 B（敵人發問攻擊）
   - 增加遊戲性和趣味性

3. **錯誤/警告**:
   - 始終使用方案 B（技能反噬）
   - 提供重試/防禦/取消選項

4. **權限請求**:
   - IF 初次請求 → 方案 C（戰前處理）
   - IF 戰鬥中請求 → 方案 B（力量借用）

## 事件類型定義

### 1. 互動事件（需用戶決策）

#### FR-1.1: Plan Mode 事件

**觸發條件**:
- Claude Code 調用 EnterPlanMode tool
- 任務複雜度超過閾值
- 需要多步驟規劃

**處理流程**:
```
戰鬥前預測 → 方案 C（戰前準備）
戰鬥中觸發 → 方案 B（戰術規劃 RPG 化）
```

**用戶操作**:
- ✅ 批准計劃 → 繼續戰鬥
- ❌ 拒絕計劃 → 重新規劃
- 🔄 修改計劃 → 調整戰術

**遊戲化元素**:
- 狀態: 敵人進入「警戒」（HP 條變黃色）
- 動畫: 魔法師思考（浮現符文）
- 音效: 思考音效、翻書音效
- 獎勵: 批准後獲得戰術 Buff（+10% 傷害，持續 3 回合）

---

#### FR-1.2: AskUserQuestion 事件

**觸發條件**:
- Claude Code 調用 AskUserQuestion tool
- 需要用戶做出選擇
- 有多個可行方案

**處理流程**:
```
敵人發動特殊技能: "困惑之問"
用戶必須回答才能繼續戰鬥
```

**用戶操作**:
- 選擇一個選項
- 或輸入自定義答案（"Other"）

**遊戲化元素**:
- 狀態: 敵人蓄力狀態（發光效果）
- 動畫: 問題浮現、選項高亮
- 音效: 發問音效、選擇音效
- 後果:
  - ✅ 正確回答 → 發現弱點（+20% 傷害）
  - ❌ 錯誤回答 → 受到反噬（-15 HP）

---

#### FR-1.3: 錯誤/警告事件

**觸發條件**:
- 工具執行失敗
- 語法錯誤
- 權限錯誤
- 網絡錯誤

**處理流程**:
```
技能施放失敗
    ↓
魔法反彈，造成反噬傷害
    ↓
提供重試/防禦/取消選項
```

**用戶操作**:
- 🔄 重試 → 再次嘗試（消耗 MP）
- 🛡️ 防禦 → 減少傷害（保留 HP）
- ❌ 取消 → 放棄操作

**遊戲化元素**:
- 傷害計算: 反噬傷害 = 錯誤嚴重度 × 5
- 連續錯誤: 每次增加 1.5 倍傷害
- 視覺: HP 條閃爍紅色、震動效果
- 音效: 爆炸音效、傷害音效

---

#### FR-1.4: 權限請求事件

**觸發條件**:
- 執行需要額外權限的操作
- Bash 危險命令（rm -rf, sudo 等）
- 修改系統文件

**處理流程**:
```
戰前初次請求 → 方案 C（戰前授權）
戰鬥中請求 → 方案 B（力量借用 RPG 化）
```

**用戶操作**:
- ✅ 授予權限 → 允許操作
- ❌ 拒絕權限 → 阻止操作
- ⏸️ 稍後決定 → 暫時跳過

**遊戲化元素**:
- 視覺: 金色光環、權限標誌
- 動畫: 光芒閃耀、力量流動
- 音效: 解鎖音效、神聖音效
- 風險提示: 用圖標表示風險等級（⚠️⚠️⚠️）

---

### 2. 工具執行事件（自動執行）

所有工具執行事件詳見 `tool-mappings.md`。

**核心原則**:
1. ✅ **每個工具都有對應的魔法** - 無論是 Bash、Read、Write 還是其他工具
2. ✅ **MP 消耗與複雜度成正比** - 簡單查詢 2-3 MP，複雜操作 10-15 MP
3. ✅ **戰鬥中的效果多樣化** - 造成傷害、獲得 Buff、發現弱點等
4. ✅ **即時反饋** - 所有操作都有視覺和遊戲化反饋
5. ✅ **失敗有代價** - 錯誤造成反噬傷害，鼓勵謹慎操作

---

## 關鍵需求

### NFR-1: 不破壞原有功能

**要求**:
- Plan Mode 和 AskUserQuestion 的核心功能必須完整保留
- 用戶的選擇和決策必須正確傳遞給 Claude Code
- 錯誤處理機制必須正常工作
- 權限系統必須安全可靠

**驗收標準**:
- ✅ 所有原有功能測試通過
- ✅ 與 Claude Code 的通信正常
- ✅ 沒有引入新的 bug

---

### NFR-2: 增強用戶體驗

**要求**:
- RPG 化包裝應該增強而非破壞用戶體驗
- 視覺設計應該清晰易懂
- 動畫效果應該流暢自然
- 音效應該恰到好處（可關閉）

**驗收標準**:
- ✅ 用戶理解事件含義（A/B 測試）
- ✅ 動畫流暢（60 FPS）
- ✅ 音效可配置（音量/開關）

---

**要求**:
