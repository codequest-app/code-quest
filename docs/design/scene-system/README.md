# 場景系統（Scene System）

**依賴**: 地圖系統
**被依賴**: 戰鬥系統、商店系統、Worktree 系統

---

## 📖 系統概要

場景系統實現雙場景架構：「探索模式」和「戰鬥模式」。不同場景有不同的 UI 風格、資源恢復規則和可用操作。系統根據用戶的 Prompt 類型智能切換場景，提供清晰的狀態區分和符合 RPG 直覺的體驗。

---

## 🎯 核心功能

### 雙場景系統

1. **探索模式（非戰鬥）**
   - 與 Claude 自由對話，不消耗 HP/MP
   - 主菜單系統：查看狀態、技能、夥伴、成就等
   - 快速資源恢復：HP +2/秒，MP +1/秒
   - Worktree 管理（公會大廳）
   - 商店訪問（商業街）

2. **戰鬥模式**
   - 回合制戰鬥界面
   - 技能施放和傷害計算
   - 慢速資源恢復：HP +0.5/秒，MP +0.1/秒
   - 戰鬥日誌和傷害顯示
   - 無法訪問商店和 Worktree 管理

### 智能場景切換

- **Prompt 分析器（PromptAnalyzer）**
  - 識別對話型：有問號 + 問題詞（什麼、為什麼、如何...）
  - 識別任務型：動作詞（生成、建立、修改、優化...）
  - 自動切換到相應場景

- **切換動畫**
  - 探索 → 戰鬥：進入戰鬥動畫（0.8秒）
  - 戰鬥 → 探索：退出戰鬥動畫（0.6秒）

## 🔗 與其他系統的關係

### 依賴關係

- **地圖系統** → 場景與地圖區域綁定
  - 城鎮區域 = 強制探索模式
  - 野外/副本 = 可能進入戰鬥模式

### 被依賴關係

- **戰鬥系統** → 在戰鬥模式下運行
- **商店系統** → 只能在探索模式訪問
- **Worktree 系統** → 只能在探索模式操作

## 💡 設計要點

### 為什麼需要雙場景？

**問題**：
- 傳統 CLI 沒有明確的「狀態」概念
- 用戶不清楚當前可以做什麼
- 資源恢復沒有差異化
- 缺乏 RPG 的節奏感

**解決方案**：
- 清晰分離探索和戰鬥狀態
- 不同場景有不同的 UI 和規則
- 提供 RPG 玩家熟悉的體驗
- 通過 Prompt 分析智能切換

### Prompt 分析邏輯

```javascript
識別對話型：
- 包含問號 "?"
- 包含問題關鍵詞（什麼、為什麼、如何、解釋、是否...）
→ 保持探索模式

識別任務型：
- 包含動作關鍵詞（生成、建立、修改、優化、重構...）
- 明確的任務指令
→ 進入戰鬥模式
```

## 🚀 快速參考

### 場景對比

| 特性 | 探索模式 | 戰鬥模式 |
|-----|---------|---------|
| **UI風格** | 休閒、菜單導向 | 緊張、戰鬥界面 |
| **對話** | 自由對話 | 戰鬥相關 |
| **HP恢復** | 2 HP/秒（快速） | 0.5 HP/秒（慢速） |
| **MP恢復** | 1 MP/秒（正常） | 0.1 MP/秒（很慢） |
| **商店** | ✅ 可訪問 | ❌ 戰鬥中無法訪問 |
| **Worktree** | ✅ 可操作 | ❌ 戰鬥中無法操作 |
| **技能** | ✅ 可使用（觸發戰鬥） | ✅ 戰鬥中使用 |

### 切換觸發

**進入戰鬥模式**：
- 用戶輸入任務型 Prompt
- 在野外/副本觸發遭遇戰
- 手動使用攻擊技能

**退出戰鬥模式**：
- 戰鬥勝利
- 戰鬥失敗
- 用戶選擇逃跑
