# Scene System - Requirements

## 核心概念

### 雙場景系統

RPG-CLI 採用雙場景系統，模擬傳統 RPG 遊戲的「探索」和「戰鬥」兩種狀態。

**設計原則**:
- ✅ 場景清晰分離，用戶明確知道當前狀態
- ✅ 不同場景有不同的 UI、規則和資源管理
- ✅ 符合傳統 RPG 玩家的直覺
- ✅ 平滑的場景切換動畫

---

## 場景 1: 探索模式（非戰鬥）

### 觸發條件

**進入探索模式**:
- 應用啟動時
- 戰鬥結束後
- 用戶主動逃跑後

**保持探索模式**:
- 用戶閒聊、詢問知識
- 呼叫菜單管理
- 查看狀態、成就等

### 核心功能

#### 1. 對話系統

**特性**:
- 與 Claude 自由對話
- 不消耗 HP/MP（知識問答）
- 不生成敵人
- 對話框樣式與戰鬥不同

**範例對話**:
- "什麼是閉包？" → 正常回答，不進入戰鬥
- "解釋一下 React Hooks" → 知識問答，不進入戰鬥
- "你好" → 打招呼，不進入戰鬥

#### 2. 主菜單系統

**菜單項目**:

| 菜單 | 圖標 | 功能 | 說明 |
|------|------|------|------|
| 狀態 | 📊 | 查看玩家狀態 | 等級、HP/MP、經驗值、統計數據 |
| 技能 | 🔮 | 管理技能 | 查看已解鎖技能、冷卻狀態 |
| 夥伴 | 🧙 | 夥伴管理 | 查看夥伴列表、夥伴詳情、設定默認夥伴 |
| 召喚獸 | 🌟 | 召喚獸圖鑑 | 查看已解鎖的召喚獸 |
| 成就 | 🏆 | 成就系統 | 查看成就進度、解鎖獎勵 |
| Worktree | 🌲 | 平行世界 | 管理 Worktree、切換世界 |
| 設定 | ⚙️ | 系統設定 | RPG 化開關、音效、視覺效果 |

#### 3. 資源恢復

**探索模式資源規則**:
- **HP 恢復**: 2 HP/秒（快速）
- **MP 恢復**: 1 MP/秒（正常）
- **技能冷卻**: 正常倒數
- **夥伴 MP**: 戰鬥結束後立即全滿

#### 4. Worktree 管理

**只能在探索模式操作**:
- ✅ 創建新世界（消耗 10 MP）
- ✅ 切換世界（消耗 5 MP）
- ✅ 合併世界（消耗 20 MP，獲得經驗值）
- ❌ 戰鬥中無法操作

## 場景 2: 戰鬥模式

### 觸發條件

**進入戰鬥模式**:
- 用戶輸入任務型 Prompt
- 系統分析複雜度 > 閾值
- 生成敵人
- 自動切換場景

**任務型 Prompt 識別**:
- 包含動作關鍵字：修復、創建、重構、實作、優化、測試、部署...
- 包含代碼相關詞：Bug、功能、API、組件、函數、類別...
- 多步驟指令
- 明確的開發任務

### 戰鬥流程

#### 1. 戰鬥開始

```
[探索模式]
用戶: "修復登入按鈕無反應的問題"
    ↓
分析複雜度: 6/10
    ↓
生成敵人: 🐛 Bug怪物 Lv.3
    ↓
[切換動畫: 畫面震動 + 敵人出現]
    ↓
[進入戰鬥模式]
```

#### 2. 回合制戰鬥

**玩家回合**:
- Claude 自動行動（使用工具 = 施放魔法）
- 用戶可以手動觸發技能
- 用戶可以召喚/切換夥伴
- 用戶可以使用道具（未來）

**敵人回合**:
- 敵人自動反擊
- 可能發動特殊技能
- 可能發問攻擊（AskUserQuestion）

**夥伴回合**（如果已召喚）:
- 夥伴 AI 自動選擇技能
- 獨立行動

#### 3. 戰鬥結束

**勝利條件**:
- 敵人 HP = 0
- 任務完成

**失敗條件**（可選，暫不實作）:
- 玩家 HP = 0
- MP 耗盡且無法行動

**戰鬥結束流程**:
```
敵人 HP = 0
    ↓
[勝利動畫]
    ↓
顯示獎勵: 經驗值 +50、金幣 +10
    ↓
經驗值累加 → 檢查是否升級
    ↓
[如果升級] 顯示升級動畫
    ↓
[切換回探索模式]
```

### 戰鬥中的行動

#### 1. 等待 Claude 行動（自動）

Claude 使用工具時，自動轉化為魔法：
- Read → 📖 讀心術
- Write → ✍️ 創造術
- Edit → ✏️ 改寫術
- Bash → 各種指令魔法
- 等等...

#### 2. 施放技能（手動）

用戶可以主動觸發技能：
- 輸入 `/commit` → 觸發版本封印術
- 輸入 `/test` → 觸發試煉之法
- 檢查 MP 和冷卻時間
- 施放成功 → 造成額外傷害

#### 3. 召喚/切換夥伴

**召喚夥伴**:
- 選擇「🧙 召喚夥伴」
- 選擇夥伴（從列表）
- 消耗 20 MP
- 夥伴加入戰鬥

**切換夥伴**（最多 2 個）:
- 選擇「🧙 切換夥伴」
- 選擇要替換的夥伴
- 消耗 10 MP

#### 4. 使用道具（未來功能）

- HP 藥水（恢復 HP）
- MP 藥水（恢復 MP）
- 特殊道具

#### 5. 防禦（可選功能）

- 降低本回合受到的傷害 50%
- 不消耗 MP

#### 6. 逃跑

- 取消當前任務
- 消耗 30% 當前 MP
- 不獲得獎勵
- 返回探索模式

### 資源規則

**戰鬥中資源**:
- **HP 恢復**: 0 HP/秒（不恢復）
- **MP 恢復**: 0.1 MP/秒（極慢）
- **技能冷卻**: 正常倒數
- **夥伴 MP**: 戰鬥中不恢復

**資源消耗**:
- 工具使用消耗 MP（2-20 MP）
- 技能施放消耗 MP（依技能）
- 夥伴召喚消耗 MP（20 MP）

## 場景切換

### 探索 → 戰鬥

**觸發條件**:
1. 用戶輸入任務型 Prompt
2. 系統識別為任務型
3. 分析複雜度 ≥ 閾值
4. 生成敵人
5. 切換到戰鬥模式

**切換時序**:
1. 畫面淡出（300ms）
2. 顯示敵人出現動畫（500ms）
3. 畫面淡入戰鬥場景（300ms）
4. 顯示「戰鬥開始」文字（1s）
5. 開始戰鬥

### 戰鬥 → 探索

**觸發條件**:
1. 敵人 HP = 0
2. 顯示勝利動畫
3. 計算和發放獎勵
4. 檢查升級
5. 夥伴 MP 全滿
6. 切換回探索模式

**切換時序**:
1. 敵人消失動畫（500ms）
2. 顯示勝利文字（1s）
3. 顯示獲得獎勵（2s）
4. 如果升級，顯示升級動畫（2s）
5. 畫面淡出（300ms）
6. 切換到探索場景
7. 畫面淡入（300ms）

### 逃跑切換

**觸發條件**:
1. 用戶選擇逃跑
2. 計算 MP 懲罰（30% 當前 MP）
3. 顯示逃跑訊息
4. 切換回探索模式

## Prompt 類型識別

### 任務型 Prompt（進入戰鬥）

**特徵**:
- 包含動作關鍵字
- 包含代碼/技術詞彙
- 多步驟指令
- 明確的目標

**範例**:
- ✅ "修復登入按鈕無反應的問題" → 戰鬥
- ✅ "創建一個用戶註冊 API" → 戰鬥
- ✅ "重構認證系統" → 戰鬥
- ✅ "幫我寫測試" → 戰鬥
- ✅ "優化這個查詢的性能" → 戰鬥

**關鍵字列表**:

動作關鍵字:
- 修復、創建、實作、開發、建立
- 重構、優化、改進、更新、升級
- 測試、部署、設定、配置、整合
- fix, create, implement, develop, build
- refactor, optimize, improve, update, upgrade
- test, deploy, setup, configure, integrate

代碼關鍵字:
- bug, 功能, api, 組件, 函數, 類別
- 代碼, 程式, 檔案, 模組, 套件
- component, function, class, code
- file, module, package, endpoint

### 對話型 Prompt（保持探索）

**特徵**:
- 知識詢問
- 解釋說明
- 一般閒聊
- 查詢資訊

**範例**:
- ❌ "什麼是閉包？" → 探索（知識問答）
- ❌ "解釋一下 React Hooks" → 探索（解釋）
- ❌ "你好" → 探索（閒聊）
- ❌ "今天天氣怎麼樣" → 探索（閒聊）
- ❌ "JavaScript 和 TypeScript 的區別" → 探索（比較）

**關鍵字列表**:

問題關鍵字:
- 什麼、為什麼、怎麼、如何、是否
- 解釋、說明、介紹、比較、區別
- what, why, how, explain, describe
- introduce, compare, difference

### 複雜度分析

即使是任務型 Prompt，也要分析複雜度決定敵人等級。

**評分因素**:

1. **長度因素**:
   - 字數越多 → 複雜度越高

2. **關鍵字數量**:
   - 多個動作關鍵字 → 複雜度越高

3. **多步驟指令**:
   - 包含"並且"、"然後" → 複雜度越高

4. **技術深度**:
   - 包含高級技術詞彙 → 複雜度越高

**範例**:
```
"修復按鈕 Bug"
→ { score: 3, level: 2 }
→ 🐛 Bug怪物 Lv.2

"重構整個認證系統並確保向後兼容"
→ { score: 15, level: 12 }
→ 🏗️ 架構惡魔 Lv.12
```

## 特殊場景

### 場景 1: 戰鬥中觸發 Plan Mode

**問題**: Plan Mode 需要用戶決策，但正在戰鬥中

**解決方案**: 暫停戰鬥，顯示「戰術規劃」事件

```
[戰鬥中]
Claude 觸發 Plan Mode
    ↓
暫停戰鬥（時間停止）
    ↓
顯示「🧙 戰術規劃」彈窗
    ↓
用戶批准/拒絕
    ↓
恢復戰鬥
```

詳見: `interactive-events/`

### 場景 2: 探索中使用 Worktree

**規則**: Worktree 只能在探索模式操作

```
[探索模式]
用戶選擇「🌲 Worktree 管理」
    ↓
顯示世界列表
    ↓
用戶選擇「切換世界」
    ↓
消耗 5 MP
    ↓
切換到新世界
    ↓
刷新畫面
```

### 場景 3: 戰鬥中嘗試 Worktree

**規則**: 戰鬥中無法操作 Worktree

```
[戰鬥模式]
用戶嘗試 Worktree 操作
    ↓
顯示錯誤訊息
    ↓
"⚠️ 戰鬥中無法切換世界！"
```

## 場景對比

### 功能差異表

| 特性 | 探索模式 | 戰鬥模式 |
|------|---------|---------|
| **目的** | 管理、休息、對話 | 完成任務、獲得獎勵 |
| **觸發** | 應用啟動、戰鬥結束 | 任務型 Prompt |
| **HP 恢復** | 2 HP/秒 | 0 HP/秒 |
| **MP 恢復** | 1 MP/秒 | 0.1 MP/秒 |
| **主要功能** | 菜單、對話、Worktree | 技能、夥伴、攻擊 |
| **可逃離** | N/A | 可以（消耗 MP） |
| **UI 風格** | 放鬆、信息豐富 | 緊湊、專注戰鬥 |

### 資源管理對比

| 資源 | 探索模式 | 戰鬥模式 |
|------|---------|---------|
| **HP 恢復速度** | 2 HP/秒（快速） | 0 HP/秒（不恢復） |
| **MP 恢復速度** | 1 MP/秒（正常） | 0.1 MP/秒（極慢） |
| **技能冷卻** | 正常倒數 | 正常倒數 |
| **夥伴 MP** | 戰鬥結束全滿 | 戰鬥中不恢復 |
| **Worktree 操作** | 允許（消耗 MP） | 禁止 |

## 設計優勢

### 1. 符合 RPG 直覺

傳統 RPG 玩家一看就懂：
- 探索 = 城鎮/安全區
- 戰鬥 = 遭遇敵人
- 明確的狀態轉換

### 2. 場景明確

用戶永遠知道當前狀態：
- UI 顏色不同
- 標題明確顯示
- 功能清楚分離

### 3. 資源管理合理

符合遊戲設計邏輯：
- 探索模式：休息恢復
- 戰鬥模式：緊張消耗
- 鼓勵策略規劃

### 4. 功能分離

不同場景不同操作：
- 探索：管理、對話
- 戰鬥：技能、攻擊
- 避免功能混亂

### 5. 擴展性強

未來可加入更多場景：
- 城鎮場景（商店、NPC）
- 地城場景（探險、寶箱）
- 特殊事件場景

---

## 總結

場景系統是 RPG-CLI 的核心架構，提供：

**✅ 清晰的狀態管理**:
- 探索 vs 戰鬥
- 不同的資源規則
- 明確的切換條件

**✅ 符合 RPG 邏輯**:
- 傳統 RPG 玩家直覺
- 安全區 vs 戰鬥區
- 休息恢復機制

**✅ 靈活的擴展性**:
- 可加入更多場景
- 可細化場景規則
- 支持複雜互動
