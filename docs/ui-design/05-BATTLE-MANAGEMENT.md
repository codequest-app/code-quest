## 🎯 設計背景與挑戰

### 核心問題

Code Quest 是一款 Dragon Quest (DQ) 風格的 RPG，強調**單一焦點、全螢幕沉浸、選單驅動**的遊戲體驗。然而，我們需要支援一個技術需求：

> **多 AI 並行戰鬥**：玩家可以同時派遣 Haiku、Sonnet、Opus 三個 AI 代理，在不同的任務/戰鬥中並行工作。

這產生了設計上的根本矛盾：

- **DQ 設計哲學**：玩家同一時間只專注於一件事（一場戰鬥、一段對話、一個選單）
- **並行需求**：技術上需要同時管理多個 AI 代理在不同戰鬥中的狀態

**設計挑戰**：如何在保持 DQ 風格純粹性的同時，提供便利的多戰鬥管理功能？

## 🔍 方案探索過程

我們探索了三種主要方案，並最終選擇了混合方案。以下是完整的決策過程：

### 方案 2：純派遣系統 (Pure Dispatch)

**核心概念**：
- 背景戰鬥完全不可見
- 玩家透過「派遣選單」選擇 AI 代理和任務
- AI 在背景工作，只在需要決策時通知玩家
- 玩家主螢幕始終保持單一焦點

**評分**：
- DQ 風格純粹性：⭐⭐⭐⭐⭐ (5/5)
- 操作便利性：⭐⭐ (2/5)

**優點**：
- 完美符合 DQ 單一焦點哲學
- 玩家永遠不會被多個戰鬥畫面干擾
- 視覺設計最簡潔、一致性最高

**缺點**：
- 派遣後完全看不到進度，缺乏掌控感
- 無法主動關注背景戰鬥的狀態
- 需要決策時的通知可能打斷主要戰鬥

**為何不採用**：
雖然 DQ 純粹性最高，但對於需要頻繁監控多個代理的用戶（如開發者調試、複雜專案管理），缺乏即時可見性會造成很大的不便。

---

### 方案 3：PIP 子視窗 (Picture-in-Picture)

**核心概念**：
- 主畫面顯示當前戰鬥（全螢幕）
- 背景戰鬥以小型 PIP 視窗顯示在角落
- PIP 視窗可拖曳、縮放
- 點擊 PIP 可切換為主畫面

**評分**：
- DQ 風格純粹性：⭐ (1/5)
- 操作便利性：⭐⭐⭐⭐ (4/5)

**優點**：
- 可同時看到所有戰鬥狀態
- 視覺資訊最豐富
- 符合現代多工 UI 習慣

**缺點**：
- **嚴重破壞 DQ 沉浸感**：同時顯示多個戰鬥畫面
- 視覺複雜度高，螢幕空間擁擠
- PIP 視窗遮擋主畫面內容
- 與 DQ「一次只做一件事」的哲學完全相反

**為何不採用**：
違背了 DQ 風格的核心設計原則。雖然功能強大，但會讓遊戲感覺像是「多任務監控系統」而非 RPG。

---

### 方案 4：分頁系統 (Tab System)

**核心概念**：
- 畫面上方有分頁列（Tab 1, Tab 2, Tab 3）
- 每個分頁顯示一場戰鬥
- 點擊分頁切換戰鬥
- 分頁上顯示簡要狀態（HP/MP/進度）

**評分**：
- DQ 風格純粹性：⭐⭐⭐ (3/5)
- 操作便利性：⭐⭐⭐⭐⭐ (5/5)

**優點**：
- 切換操作最直觀、快速
- 分頁狀態資訊一目了然
- 符合現代 UI/UX 習慣

**缺點**：
- 分頁列是**持續可見的視覺元素**，破壞 DQ 沉浸感
- 即使只有一場戰鬥，分頁列也會存在
- 不符合 DQ「選單隱藏直到呼叫」的設計語言

**為何不採用**：
雖然操作便利，但分頁列的持續存在會讓畫面感覺像是「瀏覽器」或「IDE」，而非傳統 RPG。對於單戰鬥場景（大部分時間），分頁列是不必要的視覺負擔。

---

### 混合方案：派遣 + DQ 風格快速選單 (Final Choice)

**核心概念**：
- **主要模式**：採用方案 2 的派遣系統（背景戰鬥不可見）
- **便利性補充**：加入 DQ 風格的快速選單（非持續顯示的 Tab）
- **視覺語言**：選單採用 DQ 傳統的彈出式選單設計

**評分**：
- DQ 風格純粹性：⭐⭐⭐⭐ (4/5)
- 操作便利性：⭐⭐⭐⭐ (4/5)

**設計細節**：

1. **快速選單觸發**：
   - 按下快捷鍵（例如 `Tab` 或 `Q`）
   - 彈出 DQ 風格選單（黑色半透明背景 + 金色邊框）
   - 列出所有活躍戰鬥及其狀態

2. **選單內容**：
   ```
   ┌─────────────────────────────────┐
   │ 🎯 活躍戰鬥列表                    │
   ├─────────────────────────────────┤
   │ 1. [Haiku] 實作登入功能           │
   │    HP: ████████░░ 80%            │
   │    進度: 步驟 3/5                 │
   │                                  │
   │ 2. [Sonnet] 修復支付流程           │
   │    HP: ██████████ 100%           │
   │    進度: 等待決策 ⚠️               │
   │                                  │
   │ 3. [Opus] 重構資料庫架構           │
   │    HP: ████░░░░░░ 40%            │
   │    進度: 步驟 8/12 (worktree)     │
   └─────────────────────────────────┘
   ```

3. **狀態指示**：
   - 🔵 藍色背景：一般戰鬥（直接在目錄中運作）
   - 🟣 紫色背景：複雜戰鬥（使用 worktree 隔離環境）
   - ⚠️ 警告圖示：需要玩家決策

4. **選單消失**：
   - 選擇戰鬥後，選單消失，全螢幕顯示該戰鬥
   - 按下 `ESC` 或再次按 `Tab`，選單消失，回到原戰鬥

**為何採用此方案**：

✅ **保持 DQ 沉浸感**：
- 選單只在需要時彈出，不是持續可見的 UI 元素
- 視覺設計完全符合 DQ 傳統（彈出選單、金色邊框、黑色背景）
- 主畫面始終保持單一戰鬥的全螢幕顯示

✅ **提供便利性**：
- 快捷鍵快速切換戰鬥（不比 Tab 系統慢多少）
- 選單內可看到所有戰鬥的關鍵資訊
- 警告標示讓玩家不會錯過需要決策的戰鬥

✅ **平衡的設計**：
- 不像方案 2 那樣完全看不到背景戰鬥
- 不像方案 3/4 那樣持續顯示多個戰鬥
- 在 DQ 純粹性與操作便利性之間取得良好平衡

## 🗺️ 地圖系統設計

### 雙模式設計

Code Quest 的探索系統採用**雙模式設計**，讓玩家可以根據情境選擇最適合的探索方式：

#### 模式 1：2D 俯視地圖 (Overhead Map)

**使用時機**：
- 想要探索整個世界
- 想要查看城鎮/野外的完整佈局
- 想要自由移動到不同場所

**視覺設計**：
```
┌────────────────────────────────────────┐
│          🏰 城鎮中心 (Town Square)      │
├────────────────────────────────────────┤
│                                         │
│    🏪       🍺       📚                 │
│  商業街    酒館   知識圖書館             │
│                                         │
│    🏛️       🏠       🗡️                │
│  公會大廳  玩家住所  鍛造屋              │
│                                         │
│         [玩家 👤]                       │
│                                         │
│  [方向鍵移動] [Enter 進入] [Q 選單]     │
└────────────────────────────────────────┘
```

**操作方式**：
- 方向鍵：移動角色
- Enter：進入當前場所
- Q：開啟快速選單

**特點**：
- 直觀的空間感
- 適合休閒探索
- 視覺上更有 RPG 氛圍

---

#### 模式 2：對話式選單 (Dialogue Mode)

**使用時機**：
- 明確知道要去哪裡
- 想要快速開始任務
- 不需要視覺探索的情境

**視覺設計**：
```
┌────────────────────────────────────────┐
│ 📍 你在城鎮中心。想去哪裡？              │
├────────────────────────────────────────┤
│                                         │
│  1. 🏪 商業街 - 購買道具與裝備          │
│  2. 🍺 酒館 - 接取任務與招募夥伴         │
│  3. 📚 知識圖書館 - 學習新技能           │
│  4. 🏛️ 公會大廳 - 查看成就與排行榜       │
│  5. 🏠 玩家住所 - 休息與管理夥伴         │
│  6. 🗡️ 鍛造屋 - 強化裝備與合成道具       │
│  7. 🌲 前往野外                         │
│                                         │
│  [輸入數字或直接說明你想做什麼]          │
└────────────────────────────────────────┘
```

**操作方式**：
- 輸入數字：選擇場所
- 輸入文字：自然語言描述（例如「我想買 HP 藥水」）

**特點**：
- 快速直達目的地
- 支援自然語言輸入
- 適合任務導向的玩家

---

### 模式切換

**切換方式**：
- 在地圖模式按 `M`：切換到對話模式
- 在對話模式按 `M`：切換到地圖模式
- 系統會記住玩家的偏好

**設計理由**：

1. **彈性**：不同玩家有不同偏好
2. **情境適應**：探索時用地圖，快速任務用對話
3. **學習曲線**：新手用對話（簡單），熟練後用地圖（沉浸）

- **風險提示**：當前目錄會提示可能的風險，引導玩家為複雜任務選擇 worktree
- **自動建議**：系統會根據任務複雜度自動推薦環境

#### 步驟 5：自動戰鬥設定

**自動戰鬥 AI**：

當選擇「完全背景」或「觀察模式」時，AI 會自動處理大部分決策。但某些關鍵決策仍需玩家確認。

**AI 可自動處理**：
- ✅ 選擇工具（Read, Write, Grep 等）
- ✅ 讀取檔案
- ✅ 編輯程式碼
- ✅ 執行測試
- ✅ 簡單的 git 操作（add, commit）

**需要玩家決策**：
- ❌ git push（影響遠端儲存庫）
- ❌ 刪除檔案（不可逆操作）
- ❌ 安裝新套件（改變專案依賴）
- ❌ 修改設定檔（可能影響整個專案）
- ❌ 遇到複雜錯誤（需要策略決策）

**通知方式**：
```
┌────────────────────────────────────────┐
│ ⚠️ 需要你的決策！                       │
├────────────────────────────────────────┤
│                                         │
│ 戰鬥：實作登入功能 (Haiku)              │
│                                         │
│ AI 想要執行：                           │
│ git push origin feature/login          │
│                                         │
│ 這會將變更推送到遠端儲存庫。              │
│                                         │
│ 允許？                                  │
│ [是] [否] [查看詳情]                    │
│                                         │
└────────────────────────────────────────┘
```

## 🎮 戰鬥管理架構

### 四層 UI 系統

Code Quest 的戰鬥管理採用四層架構，每層負責不同的功能：

#### 第 1 層：主戰鬥畫面 (Main Battle Screen)

**功能**：
- 顯示當前聚焦的戰鬥
- 全螢幕顯示，無其他干擾元素
- 玩家主要的互動介面

**視覺設計**：

## 🔑 關鍵設計決策說明

### 決策 1：為何選擇混合方案而非純派遣？

**問題**：為什麼不採用最純粹的 DQ 風格（方案 2：完全看不到背景戰鬥）？

**答案**：

雖然純派遣系統在 DQ 風格純粹性上得分最高，但實際使用場景顯示：

1. **開發者需求**：
   - 開發者經常需要監控 AI 的進度
   - 完全看不到會造成焦慮感（「AI 到底在做什麼？」）
   - 調試時需要即時查看 AI 的動作

2. **學習需求**：
   - 新手玩家想要學習 AI 如何解決問題
   - 即時監控可以提供教育價值

3. **信任建立**：
   - 初期用戶對 AI 的信任度較低
   - 需要透過觀察建立信任
   - 完全看不到反而會降低使用意願

4. **彈性**：
   - 混合方案提供選擇（完全背景 vs 觀察 vs 即時監控）
   - 讓不同需求的玩家都能找到適合的方式

**平衡點**：
- 預設使用派遣系統（維持 DQ 風格）
- 但提供快速選單查看進度（滿足監控需求）
- 選單採用 DQ 傳統彈出式設計（不破壞沉浸感）

---

### 決策 2：為何使用彈出選單而非 Tab 列？

**問題**：為什麼不使用更方便的分頁系統（方案 4）？

**答案**：

分頁系統雖然操作最便利，但有根本性的設計問題：

1. **持續可見性破壞沉浸感**：
   - Tab 列會一直顯示在畫面上
   - 即使只有一場戰鬥，Tab 列也存在
   - 讓遊戲感覺像「瀏覽器」或「IDE」，而非 RPG

2. **不符合 DQ 設計語言**：
   - DQ 的選單都是「隱藏直到呼叫」
   - 主選單、道具欄、裝備欄都是按鍵呼叫
   - Tab 列的持續存在違反這個原則

3. **視覺噪音**：
   - 對於大部分時間（單一戰鬥），Tab 列是不必要的
   - 增加視覺複雜度
   - 佔用螢幕空間

**彈出選單的優勢**：
- ✅ 只在需要時顯示（符合 DQ 哲學）
- ✅ 視覺設計可完全 DQ 化（黑底金框）
- ✅ 單一戰鬥時畫面完全乾淨
- ✅ 多戰鬥時快捷鍵一按即可切換

**操作便利性的補償**：
- 快捷鍵（Tab）一樣快速
- 選單顯示更豐富的資訊（不只是標題，還有 HP、進度、警告）
- 支援鍵盤數字鍵快速選擇

---

### 決策 3：為何背景戰鬥不可見？

**問題**：為什麼不讓背景戰鬥以小視窗顯示（方案 3：PIP）？

**答案**：

同時顯示多個戰鬥違反了 DQ 的核心設計哲學：

1. **單一焦點原則**：
   - DQ 設計的核心是「同一時間只專注一件事」
   - 戰鬥時就專注戰鬥，對話時就專注對話
   - 這種設計讓玩家不會被分心，體驗更沉浸

2. **視覺清晰度**：
   - PIP 視窗會遮擋主畫面
   - 多個 PIP 會讓畫面擁擠
   - 降低主要戰鬥的視覺清晰度

3. **心理負擔**：
   - 同時看到多個戰鬥會增加認知負擔
   - 玩家會不自覺地想要監控所有戰鬥
   - 反而失去了「派遣」的放鬆感

**替代方案**：
- 使用通知系統（非侵入式）
- 使用快速選單（主動查看）
- 平衡了可見性與沉浸感

---

### 決策 4：為何使用顏色區分環境？

**問題**：為什麼要用藍色和紫色區分當前目錄和 Worktree？

**答案**：

Worktree 是一個重要但容易被忽略的機制，視覺區分有助於：

1. **風險意識**：
   - 當前目錄的修改會立即影響專案
   - Worktree 的修改是隔離的
   - 顏色提醒玩家正在哪種環境工作

2. **操作提示**：
   - 紫色提醒「這是獨立環境，完成後需要合併」
   - 藍色提醒「這是主環境，小心操作」

3. **快速識別**：
   - 在快速選單中一眼看出哪些是複雜任務（紫色）
   - 幫助玩家規劃優先順序

4. **學習輔助**：
   - 新手透過顏色學習 Worktree 的概念
   - 視覺化抽象的技術概念

---

### 決策 5：為何需要雙模式地圖？

**問題**：為什麼不只提供一種探索方式（地圖或對話）？

**答案**：

不同玩家和不同情境有不同需求：

1. **玩家類型**：
   - **沉浸型玩家**：喜歡 2D 地圖，享受探索的過程
   - **效率型玩家**：喜歡對話模式，快速達成目標

2. **情境差異**：
   - **休閒探索時**：使用地圖更有趣
   - **明確目標時**：使用對話更快速
   - **學習階段**：對話模式提供引導
   - **熟練階段**：地圖模式提供自由

3. **自然語言優勢**：
   - 對話模式支援自然語言（「我想買 HP 藥水」）
   - 不需要記住精確的位置
   - 更符合 AI 助手的特性

4. **彈性設計**：
   - 讓玩家自己選擇偏好的方式
   - 不強迫所有人使用同一種模式
   - 提高整體用戶滿意度

---

### 決策 6：為何需要三種監控模式？

**問題**：為什麼不簡化為「背景」和「前景」兩種？

**答案**：

三種模式對應三種不同的信任和參與程度：

1. **完全背景**（高信任）：
   - 適合：簡單任務、信任 AI 的老手
   - 心態：「交給 AI，我去做其他事」
   - 需求：只在出問題時通知我

2. **觀察模式**（中等信任）：
   - 適合：一般任務、想要掌握進度的玩家
   - 心態：「我想知道進度，但不想一直盯著」
   - 需求：可以隨時查看，但不主動打擾

3. **即時監控**（學習模式）：
   - 適合：複雜任務、學習階段、調試
   - 心態：「我想看 AI 怎麼做，或者這任務很關鍵」
   - 需求：即時看到每個動作

**為何不合併觀察和背景**：
- 觀察模式提供「主動查看」的能力
- 但不會「被動通知」
- 這是一個重要的中間地帶

---

### 決策 7：為何自動戰鬥不完全自動？

**問題**：為什麼不讓 AI 完全自主，連 git push 也自動執行？

**答案**：

某些操作具有不可逆性或外部影響，需要玩家把關：

1. **不可逆操作**：
   - 刪除檔案：無法復原
   - 需要玩家明確同意

2. **外部影響**：
   - git push：影響遠端儲存庫和其他開發者
   - 安裝套件：改變專案依賴，可能影響團隊

3. **風險控制**：
   - 修改設定檔：可能影響整個專案運作
   - 需要玩家審查

4. **學習機會**：
   - 關鍵決策點提供教育價值
   - 幫助玩家理解 AI 的思維過程

**平衡點**：
- 90% 的操作自動化（讀、寫、測試、commit）
- 10% 的關鍵操作需要確認（push、刪除、安裝）
- 既保持便利性，又保持安全性

## 📈 與其他系統的整合

### 與場景系統的整合

**場景切換**：
- 派遣任務屬於「公會大廳」場景
- 查看戰鬥屬於「戰鬥」場景
- 切換戰鬥 = 切換場景

**地圖系統**：
- 雙模式地圖（2D / 對話）
- 從地圖進入公會大廳派遣任務
- 戰鬥完成後回到地圖

---

### 與戰鬥系統的整合

**AI 代理即戰鬥單位**：
- Haiku = 輕量級戰士（速度快）
- Sonnet = 平衡型戰士（可靠）
- Opus = 重裝戰士（強大）

**戰鬥機制**：
- HP = AI 的信心/錯誤容忍度
- MP = API 呼叫次數
- 技能 = Claude Code 工具

---

### 與 Worktree 系統的整合

**視覺整合**：
- 紫色標示 Worktree 戰鬥
- 藍色標示當前目錄戰鬥

**工作流整合**：
- 複雜任務自動建議使用 Worktree
- Worktree 戰鬥完成後提示合併
- 合併流程視覺化

---

### 與互動事件系統的整合

**決策點整合**：
- AI 的決策請求 = 互動事件
- 使用統一的通知系統
- 使用統一的決策 UI

**通知優先級**：
- 錯誤 > 需要決策 > MP 不足 > 完成

## 📝 總結

### 設計原則回顧

1. **DQ 風格優先**：所有設計都以保持 DQ 沉浸感為首要目標
2. **單一焦點**：同一時間只顯示一場戰鬥
3. **非侵入式輔助**：提供便利功能，但不破壞主體驗
4. **視覺一致性**：所有 UI 元素都符合 DQ 傳統設計語言
5. **彈性選擇**：提供多種模式，讓玩家選擇適合自己的方式

---

### 核心設計決策

| 決策 | 選擇 | 理由 |
|------|------|------|
| 多戰鬥管理方案 | 混合方案（派遣 + 快速選單） | 平衡 DQ 純粹性與操作便利性 |
| 切換機制 | 彈出選單（非 Tab 列） | 符合 DQ「選單隱藏」的設計語言 |
| 背景戰鬥可見性 | 不可見（只在選單中顯示） | 維持單一焦點原則 |
| 環境視覺區分 | 藍色（當前）vs 紫色（Worktree） | 風險意識與快速識別 |
| 探索模式 | 雙模式（2D 地圖 + 對話） | 滿足不同玩家偏好 |
| 監控模式 | 三種（背景/觀察/即時） | 對應不同信任和參與程度 |
| 自動戰鬥程度 | 部分自動（關鍵操作需確認） | 平衡便利性與安全性 |

---

### 未來可能的優化方向

1. **AI 學習**：
   - AI 從玩家的決策中學習偏好
   - 減少重複性的決策請求

2. **批次操作**：
   - 支援一次派遣多個相關任務
   - 自動規劃 AI 分配

3. **戰鬥回放**：
   - 完整記錄 AI 的思考過程
   - 支援慢速回放和逐步解析

4. **社群分享**：
   - 分享成功的戰鬥策略
   - 學習他人的 AI 使用技巧

5. **進階監控**：
   - 效能指標可視化（回應時間、錯誤率）
   - 成本分析（MP 消耗統計）
