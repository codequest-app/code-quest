# Plan Mode Screen - 計畫模式畫面

**分類**: Event & Modal Screens
**類型**: Strategic Planning Overlay
**優先級**: Critical
**最後更新**: 2026-02-05

---

## 概述

計畫模式是當異步戰鬥系統檢測到任務複雜度過高時觸發的戰略規劃介面。此模式允許 AI 制定詳細的多步驟計畫，並通過互動式 UI 讓使用者審核和批准計畫。

### 觸發條件

- 任務複雜度 > 閾值（12/15）
- 預估需要多個工作樹
- 需要複雜的依賴管理
- 用戶明確要求制定計畫

## 狀態管理

### 計畫模式狀態

```typescript
interface PlanModeState {
  isActive: boolean;
  currentStep: 1 | 2 | 3 | 4;
  plan: {
    goal: string;
    complexity: number;
    estimatedTime: number;
    estimatedCost: number;
    tasks: Task[];
    worktrees: Worktree[];
    risks: Risk[];
    expectedOutcomes: string[];
  };
  canProceed: boolean;
}
```

### 步驟驗證

- **步驟 1**: 需要定義目標和複雜度分析
- **步驟 2**: 需要至少 2 個任務
- **步驟 3**: 需要至少 1 個工作樹分配
- **步驟 4**: 所有前面步驟完成後才能到達

### 計畫執行

當用戶批准計畫後:
1. 消耗 10 MP (批准戰術成本)
2. 創建所有工作樹分支
3. 啟動第一個並行任務
4. 關閉計畫模式 modal
5. 在右側面板顯示戰鬥卡片
6. 根據依賴關係自動啟動後續任務

## 動畫效果

### 入場動畫

【參考：01-design-system/animation-timing.md】

```
0.0s  ├─ 背景遮罩淡入 (300ms)
0.1s  ├─ Modal 從中心縮放進入 (400ms, spring)
0.3s  ├─ 內容區域淡入 (300ms)
0.4s  └─ 按鈕滑入 (200ms, stagger 50ms)
```

### 步驟切換動畫

```typescript
// 前進到下一步
{
  exit: { x: -100, opacity: 0 },      // 當前內容左滑淡出
  enter: { x: 100, opacity: 0 },      // 新內容從右側進入
  animate: { x: 0, opacity: 1 }       // 滑動到中心
}

// 返回上一步
{
  exit: { x: 100, opacity: 0 },       // 當前內容右滑淡出
  enter: { x: -100, opacity: 0 },     // 新內容從左側進入
  animate: { x: 0, opacity: 1 }       // 滑動到中心
}
```

### 卡片展開/收合

```
展開: height: auto, opacity: 1 (300ms ease-out)
收合: height: 0, opacity: 0 (200ms ease-in)
```

### 批准動畫

```
0.0s  ├─ 按鈕發光脈衝 (200ms)

0.3s  ├─ 背景遮罩淡出 (200ms)
```

## 設計決策記錄

### 為什麼使用多步驟向導？

多步驟向導將複雜的計畫制定過程分解為可管理的步驟，降低認知負擔，讓使用者專注於當前階段。

### 為什麼需要審核確認步驟？

審核確認讓使用者在執行前全面了解計畫，確保計畫符合預期，避免資源浪費。

### 為什麼批准需要消耗 MP？

MP 消耗代表啟動計畫的決策成本，鼓勵使用者認真審核計畫，而不是草率批准。

### 為什麼禁用點擊外部關閉？

防止使用者在填寫過程中誤觸外部區域，導致進度丟失。需要明確的關閉操作。
