# User Question Screen - 使用者提問畫面

**分類**: Event & Modal Screens
**類型**: Interactive Question Modal
**優先級**: High
**最後更新**: 2026-02-05

---

## 概述

使用者提問畫面是當 AI 需要使用者輸入來做出決策時出現的互動式模態視窗。此畫面以 RPG 化的方式呈現問題，將 AI 的提問轉化為「敵人的發問攻擊」，使用者的回答則是「防禦反擊」。

### 觸發場景

- **技術決策**: 選擇技術棧、架構方案
- **設計選擇**: UI/UX 設計方向確認
- **衝突解決**: Git 衝突處理策略
- **風險確認**: 高風險操作前的確認
- **多選方案**: 多個可行方案需要使用者選擇

---

```typescript
interface Question {
```

## 狀態管理

### Question State

```typescript
interface QuestionState {
  isActive: boolean;
  question: Question;
  answers: Map<string, any>;  // questionId -> answer
  timeRemaining?: number;     // 剩餘秒數
  canSubmit: boolean;         // 是否可提交
}
```

### Answer Validation

- **單選題**: 至少選擇一個選項
- **多選題**: 至少選擇一個選項（除非明確允許空選）
- **文字輸入**: 非空字串
- **確認題**: 必須明確選擇 Yes 或 No

### Timeout Handling

當倒數計時結束:
1. 自動選擇預設選項（如果有）
2. 或顯示「時間到」提示
3. 禁用所有選項
4. 顯示「自動選擇」通知

## 互動設計

### 鍵盤操作

| 按鍵 | 功能 |
|------|------|
| `1-9` | 快速選擇選項 1-9 |

- **點擊選項**: 選擇/取消選擇
- **點擊卡片**: 選擇該選項並高亮
- **點擊輸入框**: 獲得焦點，開始輸入
- **雙擊選項**: 選擇並立即提交

### 觸控手勢

- **點擊選項**: 選擇
- **長按選項**: 顯示詳細說明
- **向下滑動**: 關閉 modal（需確認）

```
戰鬥畫面:
```

## 設計決策記錄

### 為什麼使用 RPG 化的問題呈現？

將問題轉化為「敵人的攻擊」增加了互動性和趣味性，讓枯燥的技術決策變得更有參與感。

### 為什麼提供快捷鍵 1-9？

快捷鍵大幅提升了鍵盤使用者的效率，特別是在需要頻繁回答問題的場景下。

### 為什麼需要「稍後決定」選項？

某些問題可能需要更多思考或查詢資料，允許用戶暫停並稍後回答可以避免倉促決策。

### 為什麼禁用點擊外部關閉？

問題通常是戰鬥的關鍵節點，誤觸外部導致關閉可能會打斷流程，需要明確的關閉操作。
