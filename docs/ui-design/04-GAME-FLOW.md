## 核心概念

### 雙場景系統

Code Quest 採用雙場景設計，模擬傳統 RPG 的探索和戰鬥狀態：

```
┌─────────────────────────────────────┐
│           場景系統                  │
├─────────────────────────────────────┤
│                                     │
│  🏰 探索模式 (Exploration)          │
│  • 城鎮區域（安全區）               │
│  • 野外區域（危險區）               │
│  • 資源恢復快速                     │
│  • 可自由移動                       │
│                                     │
│         ⇅ 轉換條件                  │
│                                     │
│  ⚔️ 戰鬥模式 (Battle)               │
│  • 任務型 Prompt 觸發               │
│  • 資源恢復緩慢                     │
│  • 專注戰鬥任務                     │
│  • 完成後返回探索                   │
│                                     │
└─────────────────────────────────────┘
```

**核心規則**：
- 探索模式：HP 恢復 2/秒，MP 恢復 1/秒
- 戰鬥模式：HP 不恢復，MP 恢復 0.1/秒
- 場景切換有明確的觸發條件和動畫

---

### 三種處理路徑

根據任務複雜度，系統自動選擇處理路徑：

```
┌─────────────────────────────────────┐
│       SmartRouter 路由決策          │
├─────────────────────────────────────┤
│                                     │
│  複雜度 0-2：對話型 (Dialog)        │
│  → 主線程直接回應                   │
│  → < 1 秒完成                       │
│  → 不消耗資源                       │
│                                     │
│  複雜度 3-7：簡單任務 (Main Sync)   │
│  → 主 CLI 同步執行                  │
│  → 幾秒到十幾秒                     │
│  → 顯示進度條                       │
│                                     │
│  複雜度 8+：複雜任務 (Battle Async) │
│  → 創建獨立戰鬥實例                 │
│  → 異步執行（可並發 3 個）          │
│  → 自動 Worktree 隔離               │
│                                     │
└─────────────────────────────────────┘
```

### 3. 緊急 Hotfix 流程

處理生產環境緊急問題的完整流程：

```
┌────────────────────────────────────────────────────┐
│            緊急 Hotfix 流程                        │
└────────────────────────────────────────────────────┘

⚠️ 階段 1：發現問題
├─ 正在進行功能開發戰鬥：
│  ├─ Worktree: feature/new-dashboard
│  ├─ 進度: 60%
│  └─ 預計還需 30 分鐘
├─ 收到緊急通知：
│  └─ "🚨 生產環境：用戶登入失敗率 100%！"
└─ 需要立即處理

🏃 階段 2：暫停當前工作
├─ 立即返回城鎮（按 B 鍵）
├─ 當前戰鬥自動暫停：
│  ├─ Worktree 保留
│  ├─ 進度保存
│  └─ 可稍後恢復
└─ 提示："戰鬥已暫停，Worktree 保持原狀"

🏛️ 階段 3：創建 Hotfix Worktree
├─ 前往公會大廳
├─ 點擊「創建新時間線」
├─ 選擇類型：Hotfix（緊急修復）
├─ 設定：
│  ├─ 名稱：hotfix/login-crash
│  ├─ 基於：main（而非開發分支！）
│  ├─ 優先級：🔴 緊急
│  └─ 預估時間：15-30 分鐘
├─ 創建成功
└─ 消耗：10 MP

⚔️ 階段 4：執行 Hotfix 戰鬥
├─ 前往野外
├─ 輸入："修復登入崩潰問題，檢查 auth.ts 的 JWT 驗證"
├─ 複雜度：9（中等）
├─ 生成敵人：🐛 緊急 Bug Lv.9
├─ 在 hotfix/login-crash Worktree 中戰鬥
├─ 戰鬥過程：
│  ├─ 使用 Read 檢查日誌
│  ├─ 發現問題：過期的 JWT secret
│  ├─ 使用 Edit 修復
│  ├─ 使用 Test 驗證
│  └─ 15 分鐘完成
└─ 戰鬥勝利

✅ 階段 5：立即合併和部署
├─ 戰鬥結束，選擇「立即合併」
├─ 合併流程：
│  ├─ Hotfix → main
│  ├─ 運行測試：✅ 通過
│  ├─ 創建 commit："hotfix: 修復登入崩潰"
│  └─ 推送到 production
├─ 部署成功
├─ 清理 Worktree：hotfix/login-crash
└─ 獲得獎勵：EXP +60, Gold +40

🔙 階段 6：恢復原工作
├─ 返回公會大廳
├─ 查看 Worktree 列表：
│  └─ feature/new-dashboard（暫停中）
├─ 點擊「恢復戰鬥」
├─ 切換回該 Worktree

### Prompt 類型識別

系統如何判斷 Prompt 類型：

```
┌────────────────────────────────────────────────────┐
│          Prompt 類型識別演算法                     │
└────────────────────────────────────────────────────┘

function analyzePromptType(prompt) {
  // 1. 檢查長度
  const length = prompt.length;

  // 2. 檢查關鍵字
  const taskKeywords = [
    '修復', '創建', '實作', '開發', '建立',
    '重構', '優化', '改進', '更新', '升級',
    '測試', '部署', '設定', '配置', '整合',
    'fix', 'create', 'implement', 'develop',
    'refactor', 'optimize', 'improve'
  ];

  const dialogKeywords = [
    '什麼', '為什麼', '怎麼', '如何', '是否',
    '解釋', '說明', '介紹', '比較', '區別',
    'what', 'why', 'how', 'explain', 'describe'
  ];

  const codeKeywords = [
    'bug', '功能', 'api', '組件', '函數',
    '代碼', '程式', '檔案', '模組',
    'component', 'function', 'class', 'code', 'file'
  ];

  // 3. 計分
  let taskScore = 0;
  let dialogScore = 0;

  // 檢查任務關鍵字
  taskKeywords.forEach(keyword => {
    if (prompt.includes(keyword)) taskScore += 2;
  });

  // 檢查代碼關鍵字
  codeKeywords.forEach(keyword => {
    if (prompt.includes(keyword)) taskScore += 1;
  });

  // 檢查對話關鍵字
  dialogKeywords.forEach(keyword => {
    if (prompt.includes(keyword)) dialogScore += 2;
  });

  // 4. 判斷類型
  if (dialogScore > taskScore) {
    return 'dialog';  // 對話型
  } else if (taskScore > 0) {
    return 'task';    // 任務型
  } else {
    return 'dialog';  // 默認對話型
  }
}
```

### 複雜度計算

詳細的複雜度評分演算法：

```
function analyzeComplexity(prompt) {
  let score = 0;

  // 1. 長度因素 (0-3 分)
  const length = prompt.length;
  if (length < 100) {
    score += 1;
  } else if (length < 200) {
    score += 2;
  } else {
    score += 3;
  }

  // 2. 關鍵字權重
  const heavyKeywords = [
    '重構', '遷移', '升級', '整個', '所有',
    '完整', '系統', '架構',
    'refactor', 'migrate', 'entire', 'whole', 'architecture'
  ];
  const mediumKeywords = [
    '創建', '實作', '開發', '優化', '設計', '集成',
    'create', 'implement', 'develop', 'optimize'
  ];
  const lightKeywords = [
    '修復', '更新', '檢查', '查看', '顯示',
    'fix', 'update', 'check', 'view'
  ];

  heavyKeywords.forEach(kw => {
    if (prompt.includes(kw)) score += 3;
  });
  mediumKeywords.forEach(kw => {
    if (prompt.includes(kw)) score += 2;
  });
  lightKeywords.forEach(kw => {
    if (prompt.includes(kw)) score += 1;
  });

  // 3. 文件數量 (0-3 分)
  const multiFileIndicators = [
    '多個', '所有', '整個', '完整',
    'multiple', 'all', 'entire'
  ];
  const hasMultiFile = multiFileIndicators.some(
    ind => prompt.includes(ind)
  );
  if (hasMultiFile) {
    score += 3;
  } else {
    score += 1;
  }

  // 4. 工具複雜度 (0-3 分)
  const complexTools = [
    'plan mode', '計劃', '設計',
    'test', '測試', 'deploy', '部署'
  ];
  complexTools.forEach(tool => {
    if (prompt.toLowerCase().includes(tool)) {
      score += 2;
    }
  });

  // 5. 依賴關係 (0-2 分)
  const dependencyIndicators = [
    '整合', '連接', '依賴', '關聯',
    'integrate', 'connect', 'dependency'
  ];
  dependencyIndicators.forEach(ind => {
    if (prompt.includes(ind)) score += 2;
  });

  // 限制最大值
  return Math.min(score, 15);
}

function generateEnemy(prompt, complexity) {
  // 1. 確定敵人類型（基於關鍵字）
  const enemyTypes = {
    'bug': { icon: '🐛', name: 'Bug', multiplier: 1.0 },
    'feature': { icon: '🏗️', name: '功能', multiplier: 1.2 },
    'refactor': { icon: '🔄', name: '重構', multiplier: 1.5 },
    'security': { icon: '🔒', name: '安全', multiplier: 1.3 },
    'performance': { icon: '⚡', name: '性能', multiplier: 1.4 },
    'architecture': { icon: '🏛️', name: '架構', multiplier: 1.8 },
    'test': { icon: '🧪', name: '測試', multiplier: 1.1 }
  };

  // 2. 匹配敵人類型
  let type = null;
  if (prompt.includes('bug') || prompt.includes('修復')) {
    type = enemyTypes.bug;
  } else if (prompt.includes('重構')) {
    type = enemyTypes.refactor;
  } else if (prompt.includes('架構')) {
    type = enemyTypes.architecture;
  } else if (prompt.includes('安全')) {
    type = enemyTypes.security;
  } else if (prompt.includes('測試')) {
    type = enemyTypes.test;
  } else {
    type = enemyTypes.feature;  // 默認
  }

  // 3. 生成敵人名稱
  const suffixes = {
    low: ['小怪', '嘍囉', '學徒'],
    medium: ['魔物', '守護者', '戰士'],
    high: ['惡魔', '巨獸', '霸主'],
    boss: ['魔王', '君王', '主宰']
  };

  let suffix;
  if (complexity <= 5) suffix = suffixes.low[0];
  else if (complexity <= 10) suffix = suffixes.medium[Math.floor(Math.random() * 3)];
  else if (complexity <= 13) suffix = suffixes.high[Math.floor(Math.random() * 3)];
  else suffix = suffixes.boss[Math.floor(Math.random() * 3)];

  const enemyName = `${type.icon} ${type.name}${suffix}`;

  // 4. 計算屬性
  const level = complexity;
  const maxHp = level * 100 * type.multiplier;
  const attack = level * 10;
  const defense = level * 5;

  return {
    name: enemyName,
    level,
    maxHp,
    hp: maxHp,
    attack,
    defense,
    type: type.name,
    icon: type.icon
  };
}

│  ├─ 運行測試驗證
│  ├─ 刪除 Worktree
│  │  └─ git worktree remove ../worktrees/user-login
│  └─ 刪除分支（可選）
├─ 獲得合併獎勵：EXP +20, Gold +10
└─ 提示："✅ 已合併到 main，Worktree 已清理"

選項 B：保留分支
├─ 執行流程：
│  ├─ Worktree 保留
│  ├─ 分支保留
│  └─ 在公會大廳可見
├─ 用戶可稍後：
│  ├─ 繼續在該分支工作
│  ├─ Code Review 後合併
│  └─ 或刪除
└─ 提示："💾 分支已保留：feature/user-login"

選項 C：放棄成果
├─ 執行流程：
│  ├─ 刪除 Worktree
│  ├─ 刪除分支
│  └─ 所有變更丟失
├─ 確認對話框（防誤操作）
└─ 提示："🗑️ 已放棄並清理 Worktree"
```

### 手動 Worktree 管理

用戶在公會大廳手動管理 Worktree：

```
┌────────────────────────────────────────────────────┐
│           手動 Worktree 管理流程                   │
└────────────────────────────────────────────────────┘

操作 1：創建新 Worktree
├─ 在公會大廳點擊「創建新時間線」
├─ 填寫表單：
│  ├─ 類型：Feature / Fix / Hotfix / Experiment
│  ├─ 名稱：feature/new-ui
│  ├─ 基於分支：main ▼
│  └─ 描述：重新設計用戶界面
├─ 消耗 10 MP
├─ 執行創建：
│  ├─ git worktree add ../worktrees/new-ui -b feature/new-ui
│  └─ 創建成功
└─ 選擇：[立即切換] or [稍後]

操作 2：切換 Worktree
├─ 在公會大廳查看 Worktree 列表
├─ 點擊某個 Worktree
├─ 選擇「進入」或「切換」
├─ 消耗 5 MP
├─ 執行切換：
│  ├─ 保存當前狀態
│  ├─ 切換到目標 Worktree 目錄
│  └─ 刷新界面
└─ 提示："已切換到：feature/new-ui"

操作 3：合併 Worktree
├─ 選擇要合併的 Worktree
├─ 點擊「合併」
├─ 顯示變更預覽：
│  ├─ Diff 摘要
│  ├─ 受影響文件列表
│  └─ 衝突檢測
├─ 確認合併
├─ 執行流程：
│  ├─ 切換到 main
│  ├─ git merge feature/new-ui
│  ├─ 運行測試
│  ├─ [如果成功] 完成合併
│  └─ [如果失敗] 顯示錯誤，回滾
├─ 消耗 20 MP
├─ 獲得獎勵
└─ 選擇是否刪除 Worktree

操作 4：刪除 Worktree
├─ 選擇要刪除的 Worktree
├─ 點擊「刪除」
├─ 確認對話框（警告會丟失未提交的變更）
├─ 執行刪除：
│  ├─ git worktree remove ../worktrees/new-ui
│  ├─ [可選] git branch -D feature/new-ui
│  └─ 從列表移除
└─ 提示："已刪除 Worktree：feature/new-ui"

操作 5：清理建議
├─ 公會大廳顯示「🧹 智能清理」按鈕
├─ 點擊後分析：
│  ├─ 已合併但未清理的 Worktrees
│  ├─ 長時間閒置的 Worktrees
│  └─ 孤立的 Worktrees（分支已刪除）
├─ 顯示清理建議列表
├─ 用戶勾選要清理的項目
├─ 批量清理
└─ 釋放磁盤空間
```

## 總結

本文件涵蓋了 Code Quest 的完整遊戲流程：

1. **使用者旅程**：新手教學、日常開發、緊急修復
2. **場景轉換**：探索 ↔ 戰鬥，場所切換，動畫效果
3. **狀態機**：完整的遊戲狀態定義和轉換規則
4. **遭遇戰機制**：Prompt 識別、複雜度計算、敵人生成
5. **互動事件**：Plan Mode、AskUserQuestion、錯誤處理
6. **Worktree 工作流**：自動創建、手動管理、合併策略
7. **多戰鬥並行**：並發管理、隊列機制、實際場景

這些流程確保用戶體驗流暢、直觀，並充分利用 Worktree 實現安全的並發開發。
