## 系統概述

### 核心概念

將 Subagent 從單純的「召喚造成傷害」提升為「戰鬥夥伴」，擁有獨立 HP/MP、專屬技能、AI 行為。

**Subagent → 戰鬥夥伴映射**：

| Subagent 特性 | 戰鬥夥伴呈現 |
|--------------|------------|
| Agent 類型 | 夥伴職業/角色 |
| Agent Memory | 夥伴經驗/成長 |
| Agent Model (sonnet/opus) | 夥伴等級/能力 |
| Agent Tools | 夥伴專屬技能 |
| 持續執行 | 戰場上的持續存在 |
| Agent 完成任務 | 施放技能/攻擊 |

```
夥伴系統 (L2 - 擴展層)

└─ 被依賴於：
   └─ 召喚獸系統 (協同效果)
```

## 核心規則

### 規則 1：夥伴狀態與屬性

#### 基本資訊

```
夥伴基本資料：
├─ agentName: Agent ID
├─ characterName: 角色名稱（如 "CodeGuard"）
├─ title: 稱號（如 "代碼守護者"）
└─ avatar: 頭像 emoji
```

---

#### 戰鬥屬性

```
屬性系統：
├─ level: 等級（基於使用次數）
├─ hp/maxHp: 生命值
├─ mp/maxMp: 魔力值（用於施放技能）
├─ attack: 攻擊力
├─ defense: 防禦力
├─ speed: 速度（決定行動順序）
└─ wisdom: 智慧（影響技能效果）
```

---

#### 狀態追蹤

```
狀態類型：
├─ standby: 待命中
├─ active: 戰鬥中
├─ exhausted: 耗盡
└─ ko: 倒下
```

---

#### 技能與成長

```
成長數據：
├─ skills: 專屬技能列表
├─ currentCooldowns: 技能冷卻狀態
├─ experience: 夥伴經驗值
├─ expToNextLevel: 升級所需經驗（100 * 1.3^(level-1)）
├─ timesDeployed: 出戰次數
└─ battlesWon: 勝利次數
```

---

### 規則 2：夥伴專屬技能

每個 Agent 根據其特性擁有專屬戰鬥技能。

#### 技能類型

```
三大類型：
├─ attack: 攻擊技能（造成傷害）
├─ support: 支援技能（治療、護盾、Buff）
└─ special: 終極技能（強力效果，冷卻長）
```

---

#### 技能屬性

```
技能資料結構:
• 技能ID (唯一識別碼)
• 技能名稱
• 圖標
• 技能類型
• MP 消耗
• 冷卻回合數
• 技能描述
• 技能效果
• 動畫名稱
• 敵人相性效果倍率
• 解鎖等級需求 (選擇性)
```

---

#### 範例：CodeGuard (code-guardian)

**技能 1: 安全掃描**
```
類型: 攻擊
MP 消耗: 20
冷卻: 2 回合
效果: 造成 60 + wisdom * 0.8 傷害
額外效果: 降低敵人防禦 -10（持續 2 回合）
相性: bug-hunt ×1.5, code-task ×1.2
```

**技能 2: 守護之盾**
```
類型: 支援
MP 消耗: 30
冷卻: 3 回合
效果: 為玩家提供 50 點護盾（持續 3 回合）
額外效果: 玩家防禦 +20（持續 2 回合）
```

**技能 3: OWASP 打擊**
```
類型: 終極技
MP 消耗: 50
冷卻: 5 回合
效果: 造成 150 + wisdom * 1.5 傷害
特殊: 對 bug-hunt 類型敵人傷害 ×2.0
需求: Lv.5
```

---

### 規則 3：被動技能

夥伴擁有被動技能，無需手動觸發。

#### 觸發類型

```
被動觸發：
├─ on_player_attacked: 玩家受攻擊時
├─ on_enemy_damaged: 敵人受傷時
├─ on_battle_start: 戰鬥開始時
└─ passive: 永久生效
```

---

#### 範例：CodeGuard 被動技能

**警戒**
```
觸發: 玩家受攻擊時
效果: 30% 機率阻擋攻擊
說明: 保護玩家免受傷害
```

**代碼洞察**
```
觸發: 永久被動
效果: 對 code-task 和 bug-hunt 類型敵人額外 +20% 傷害
說明: 專精於代碼相關戰鬥
```

---

### 規則 4：資源管理

#### MP 系統

**戰鬥中**:
```
戰鬥規則：
├─ regenInBattle: 0（戰鬥中不自動恢復）
├─ 消耗: 使用技能扣除對應 MP
└─ 初始 MP: 等於 maxMp
```

**戰鬥結束**:
```
重置機制：
├─ resetOnBattleEnd: true（完全恢復）
├─ HP 也完全恢復
└─ 清除所有冷卻
```

**設計理由**:
- 簡化戰鬥系統，避免過多變數
- 夥伴作為支援角色，在戰鬥外自動準備完成
- 玩家只需關注技能選擇，不需管理夥伴 MP

---

#### 召喚成本

**玩家召喚夥伴需消耗**:
```
成本設定：
├─ MP: 20-30（依夥伴稀有度）
└─ 冷卻: 200-300 秒（防止頻繁召喚）
```

**召喚限制**:
```
限制規則：
├─ 同時最多 2 個夥伴在場
├─ 同一夥伴同時只能存在一個實例
└─ 戰鬥中可召喚/切換夥伴
```

---

### 規則 5：AI 系統

#### 夥伴 AI 行為

夥伴在戰鬥中可以自動行動或由玩家控制。

**AI 決策優先級**:

```
決策樹：
1. 【玩家生命危險】（HP < 30%）
   └─ 優先使用支援技能（護盾、治療）

2. 【敵人類型相性】
   └─ 選擇對敵人有效的技能（effectiveness ≥ 1.2）

3. 【預設攻擊】
   └─ 使用可用的攻擊技能
   └─ 如無技能可用，執行普通攻擊
```

---

#### 技能選擇邏輯

```
技能選擇流程:

1. 過濾可用技能:
   • 檢查技能是否在冷卻中
   • 檢查MP是否足夠

2. 計算預期收益:
   • 評估每個技能的預期效果
   • 考慮敵人相性、當前戰況

3. 選擇最佳技能:
   • 按收益排序
   • 選擇收益最高的技能
```

---

#### 夥伴個性

每個夥伴有不同的 AI 風格：

```
三種風格：
├─ defensive: 優先保護玩家
├─ aggressive: 優先輸出傷害
└─ balanced: 平衡攻防
```

---

### 規則 6：夥伴成長系統

#### 經驗值獲取

夥伴獲得經驗的來源：

| 來源 | 經驗值 |
|------|--------|
| 戰鬥勝利 | 50 |
| 使用技能 | 5 |
| 造成傷害 | damage / 10 |
| 治療玩家 | amount / 5 |
| 命中弱點 | 20 |

---

#### 升級機制

**經驗值需求**:
```
升級所需經驗 = 基礎值 × (成長率 ^ (等級 - 1))
```

夥伴升級比玩家快（玩家成長率 1.5，夥伴成長率 1.3）。

**示例**：
```
Lv.1 → Lv.2: 100 EXP
Lv.2 → Lv.3: 130 EXP
Lv.3 → Lv.4: 169 EXP
Lv.4 → Lv.5: 220 EXP
```

---

#### 升級獎勵

每次升級獲得：

```
屬性提升：
├─ maxHp: +20
├─ maxMp: +15
├─ attack: +5
├─ defense: +5
├─ speed: +2
└─ wisdom: +5
```

---

#### 特殊等級解鎖

```
里程碑：
├─ Lv.5: 解鎖終極技能
├─ Lv.10: 解鎖額外被動技能
└─ Lv.15: 屬性大幅提升
```

---

### 規則 7：多夥伴系統

#### 召喚規則

**最大同時數量**: 2 個夥伴

**檢查召喚條件**:
```
檢查清單：
1. 當前夥伴數 < 2
2. 玩家 MP 足夠
3. 夥伴不在冷卻中
4. 該夥伴尚未在場
```

**切換夥伴**:
```
切換機制：
├─ 消耗: 10 MP
├─ 可在戰鬥中執行
├─ 舊夥伴返回待命
└─ 新夥伴加入
```

**解除召喚**:
```
解除機制：
├─ 無消耗
├─ 夥伴返回待命狀態
└─ 釋放夥伴槽位
```

---

#### 回合順序

**行動順序計算**:
基於所有單位的速度值排序：

```
行動順序 = [玩家, 夥伴1, 夥伴2, 敵人] (按速度排序)
```

**速度影響**:
- 速度高的單位先行動
- 速度相同時，玩家 > 夥伴 > 敵人

---

#### 回合流程

```
回合開始
    ↓
玩家回合
    ↓
夥伴1回合（自動）
    ↓
夥伴2回合（自動）
    ↓
敵人回合
    ↓
檢查戰鬥結束
    ↓
更新冷卻
    ↓
下一回合
```

## 系統整合

### 與 Battle System 整合

```
修改 BattleManager：
├─ 添加夥伴管理（Map<agentId, BattleCompanion>）
├─ 添加回合順序計算
├─ 添加夥伴回合執行
└─ 添加夥伴傷害計算

戰鬥流程變化：
原始流程: 玩家 → 敵人 → 檢查結束
新流程: 玩家 → 夥伴1 → 夥伴2 → 敵人 → 檢查結束
```

---

### 敵人 AI 變化

```
AI 等級調整：
├─ simple AI: 只攻擊玩家，忽略夥伴
├─ elite AI: 30% 機率攻擊夥伴
└─ boss AI: 50% 機率攻擊夥伴，可使用 AOE
```

---

### 與 Summon Beast System 整合

```
協同效果：
CodeGuard + Code Dragon = 代碼雙龍陣
├─ 條件: 夥伴和召喚獸同時在場
├─ 效果: damage_multiplier ×1.5, defense_boost +30
└─ 描述: 防禦和攻擊大幅提升
```

## 設計決策

### 決策 1：為什麼需要夥伴系統？

**問題**：
- Subagent 只是「一次性召喚」
- 缺乏成長感
- 沒有情感連結

**解決方案**：
- 夥伴 = 長期戰友
- 擁有成長系統
- 有個性和技能

**好處**：
- ✅ 更深的遊戲性
- ✅ 養成樂趣
- ✅ 情感連結

---

### 決策 2：為什麼戰鬥後 MP 全滿？

**設計理由**：
- 簡化管理
- 降低複雜度
- 專注戰鬥策略

**好處**：
- ✅ 用戶無需管理夥伴資源
- ✅ 每場戰鬥狀態一致
- ✅ 降低學習曲線

---

### 決策 3：為什麼最多 2 個夥伴？

**平衡考量**：
- 太少（1個）：選擇性不足
- 太多（3+個）：畫面混亂，操作複雜

**2 個的優勢**：
- ✅ 策略選擇（坦克+輸出，或雙輸出）
- ✅ 畫面清晰
- ✅ 回合不過長

---

### 決策 4：AI 自動行動 vs 手動控制

**選擇自動**：
- 降低操作負擔
- 夥伴 AI 足夠智能
- 保持戰鬥流暢

**未來可選**：
- 提供「手動模式」開關
- 讓玩家選擇控制方式

---

### 決策 5：夥伴升級比玩家快

**公式對比**：
```
玩家: expToNextLevel = 100 * (1.5 ^ (level - 1))
夥伴: expToNextLevel = 100 * (1.3 ^ (level - 1))
```

**設計理由**：
- 夥伴使用頻率可能較低
- 快速升級增加成就感
- 鼓勵使用不同夥伴

---

## 總結

將 Subagent 設計為戰鬥夥伴提供：

**戰略深度**:
- 多樣化職能選擇
- 技能組合搭配
- AI 自動化支援

**成長系統**:
- 經驗值累積
- 等級提升
- 技能解鎖

**RPG 體驗**:
- 夥伴養成
- 情感連結
- 協同戰鬥

**AI 整合**:
- Agent → 夥伴映射
- 自然的功能結合
- 沉浸式互動
