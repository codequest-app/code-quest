# L1 戰鬥系統

**版本**: v2.0
**更新日期**: 2026-02-06

---

## 一、系統概述

戰鬥系統是 Code Quest 的核心體驗引擎，負責將 AI 對話過程**視覺化為 RPG 回合制戰鬥**。用戶的任務型 Prompt 生成敵人，Claude 使用的工具映射為魔法技能，任務完成等同於擊敗敵人。

在系統架構中，戰鬥系統位於 **L1 核心戰鬥層**，是連接基礎層（L0）與戰鬥擴展層（L2）的樞紐。被 5 個上層系統依賴，是架構中被依賴最多的系統。

**關鍵原則**：戰鬥只是**視覺包裝** — 不改變 Claude Code 的核心 AI 功能。Claude 正常處理任務，工具使用自動映射為攻擊動畫，用戶無需任何額外操作。

## 二、依賴關係

```
依賴：
  └─ 場景系統 (2) — 場景切換到戰鬥模式時啟動戰鬥

被依賴：
  ├─ 夥伴系統 (4)      — 夥伴在戰鬥中提供持續支援
  ├─ 召喚獸系統 (5)    — 召喚獸在戰鬥中提供爆發支援
  ├─ 互動事件系統 (7)  — 戰鬥中的特殊事件（Plan Mode、錯誤等）
  ├─ 非同步戰鬥系統 (9) — 多場戰鬥並行管理
  └─ 多模型整合系統 (10) — 不同模型在戰鬥中的表現差異
```

| 方向 | 系統 | 關係說明 |
|------|------|---------|
| 依賴 | 場景系統 (2) | 場景切換為戰鬥模式時，戰鬥系統啟動 |
| 被依賴 | 夥伴系統 (4) | 夥伴在戰鬥回合中獨立行動 |
| 被依賴 | 召喚獸系統 (5) | 召喚獸在戰鬥中提供 1-3 回合的爆發輸出 |
| 被依賴 | 互動事件系統 (7) | 戰鬥中觸發的 Plan Mode、錯誤等事件 |
| 被依賴 | 非同步戰鬥系統 (9) | 管理多個戰鬥實例的並行執行 |
| 被依賴 | 多模型整合系統 (10) | 不同 AI 模型的戰鬥能力差異 |

## 三、核心規則

### 3.1 敵人生成

根據 Prompt 的複雜度和所在區域自動生成敵人：

**敵人屬性計算**：

```
敵人 HP    = 50 × 複雜度等級
敵人等級   = 複雜度值
敵人屬性   = 基於 Prompt 主題自動判定
```

**主題→敵人類型映射**：

| Prompt 主題 | 敵人類型 | 屬性 | 弱點 |
|------------|---------|------|------|
| Bug 修復 | 蟲類（Bug 系列） | 火 | 水（分析型技能） |
| 重構 / 架構 | 結構體（建築系列） | 木 | 火（攻擊型技能） |
| 效能優化 | 性能獸（速度系列） | 水 | 木（建構型技能） |
| 安全修復 | 暗影（安全系列） | 火 | 木（建構型技能） |
| 測試 | 測試幻象（幻象系列） | 水 | 火（攻擊型技能） |
| 部署 | 部署守衛（機械系列） | 木 | 水（分析型技能） |

**敵人 HP 範例**：

```
複雜度 5  → HP 250  （森林小怪）
複雜度 8  → HP 400  （山脈中敵）
複雜度 10 → HP 500  （荒原強敵）
複雜度 15 → HP 750  （火山精英）
複雜度 20 → HP 1000 （副本 Boss）
```

### 3.2 回合制戰鬥

每個回合按固定順序執行三個階段：

```
回合開始
    │
    ▼
┌──────────────┐
│ 1. 玩家回合   │  Claude 的工具使用 = 自動施放魔法
│              │  用戶可手動觸發額外技能
│              │  用戶可召喚/切換夥伴
└──────┬───────┘
       ▼
┌──────────────┐
│ 2. 夥伴回合   │  夥伴 AI 自動選擇最佳技能
│  （如有夥伴） │  優先級：治療 > 輔助 > 攻擊
└──────┬───────┘
       ▼
┌──────────────┐
│ 3. 敵人回合   │  敵人自動反擊
│              │  可能觸發特殊攻擊
│              │  可能觸發互動事件
└──────┬───────┘
       ▼
    下一回合...
```

### 3.3 傷害計算公式

```
傷害 = 技能基礎威力 × 屬性相性倍率 × (1 + 隨機浮動)

其中：
  技能基礎威力 = 每個技能的固定值
  屬性相性倍率 = 見下方相性表
  隨機浮動     = -10% ~ +10%（即 0.9 ~ 1.1）
```

**屬性相性表**：

| 攻擊屬性 | 克制（×1.5 倍傷害） | 被克制（×0.5 倍傷害） |
|---------|-------------------|---------------------|
| 火（攻擊型：Write, Bash） | 木（結構類敵人） | 水（邏輯類敵人） |
| 水（分析型：Read, Grep） | 火（Bug 類敵人） | 木（資料類敵人） |
| 木（建構型：Edit, Glob） | 水（性能類敵人） | 火（安全類敵人） |
| 雷（組合型：並行操作） | 所有屬性 ×1.2 | 無弱點 |

### 3.4 工具→魔法映射

Claude Code 的工具使用自動映射為 RPG 魔法：

| Claude 工具 | RPG 魔法名稱 | MP 消耗 | 基礎威力 | 屬性 |
|------------|-------------|---------|---------|------|
| Read | 讀心術 | 2 MP | 10 | 水 |
| Write | 創造術 | 5 MP | 25 | 火 |
| Edit | 改寫術 | 3 MP | 15 | 木 |
| Bash | 系統魔法 | 5 MP | 30 | 火 |
| Grep | 偵測術 | 2 MP | 10 | 水 |
| Glob | 探索術 | 2 MP | 10 | 木 |

**自動戰鬥機制**：Claude 處理任務時自然使用工具，每次工具調用自動映射為一次魔法攻擊。用戶無需手動操作，觀看即可。

### 3.5 戰鬥結果

**勝利條件**：敵人 HP = 0

```
勝利獎勵計算：
  EXP = 基礎 EXP × 敵人等級倍率
  Gold = 基礎 Gold × 敵人等級倍率

  基礎 EXP  = 50
  基礎 Gold = 20
  等級倍率  = 敵人等級 × 0.5
```

**勝利獎勵範例**：

```
敵人等級 5  → EXP 125, Gold 50
敵人等級 10 → EXP 250, Gold 100
敵人等級 15 → EXP 375, Gold 150
敵人等級 20 → EXP 500, Gold 200
```

**逃跑**：消耗 30% 當前 MP，不獲得獎勵。

### 3.6 Worktree 自動建立

每場戰鬥開始時自動建立獨立的 Worktree 分支：

```
戰鬥觸發
    │
    ▼
自動建立 Worktree（戰鬥分支）
    │
    ▼
戰鬥在獨立分支中進行
    │
    ▼
戰鬥結束
    │
    ├─ 勝利 → 提示：合併 / 保留 / 放棄
    └─ 逃跑 → 分支保留（可稍後在公會大廳管理）
```

## 五、與其他系統的互動

### 5.1 戰鬥 ← 場景系統 (2)

場景系統在滿足條件時觸發戰鬥：

```
場景系統                        戰鬥系統
   │                               │
   │  戰鬥觸發                      │
   │  (敵人資訊 + 區域 + 複雜度)     │
   │──────────────────────────────►│
   │                               │  生成敵人
   │                               │  建立 Worktree
   │                               │  開始回合制
   │                               │
   │  戰鬥結束                      │
   │  (勝利/逃跑 + 獎勵)            │
   │◄──────────────────────────────│
   │                               │
   │  切回探索模式                   │
```

### 5.2 戰鬥 → 夥伴系統 (4)

戰鬥中每回合的夥伴行動階段：

- 戰鬥系統通知夥伴系統「輪到夥伴行動」
- 夥伴系統的 AI 選擇技能並執行
- 傷害/治療結果回傳戰鬥系統更新戰場狀態

### 5.3 戰鬥 → 召喚獸系統 (5)

玩家在戰鬥中使用召喚獸：

- 戰鬥系統觸發召喚獸的出場動畫
- 召喚獸執行 1-3 回合的強力攻擊
- 效果結束後自動退場，回歸戰鬥系統正常流程

### 5.4 戰鬥 → 互動事件系統 (7)

戰鬥進行中可能觸發的特殊事件：

| 事件 | 觸發條件 | 處理方式 |
|------|---------|---------|
| Plan Mode | AI 判斷需要深度規劃 | 暫停戰鬥，進入靜止之間 |
| AskUserQuestion | AI 需要用戶確認 | 彈出對話，回答後繼續 |
| 錯誤/警告 | 工具執行出錯 | 呈現為敵人特殊攻擊（反噬傷害） |
| 權限請求 | 需要用戶授權 | 彈出確認，授權後繼續 |

### 5.5 戰鬥 → 非同步戰鬥系統 (9)

非同步戰鬥系統管理多個戰鬥實例：

- 最多 3 個並發戰鬥
- 每個戰鬥有獨立的 Worktree
- 用戶可在戰鬥間切換

### 5.6 戰鬥 → 多模型整合系統 (10)

不同 AI 模型影響戰鬥表現：

- 模型能力差異映射為不同的攻擊力/精準度
- 模型成本映射為 MP 消耗差異

## 六、設計決策說明

### 6.1 為什麼戰鬥只是「視覺包裝」？

**問題**：如果戰鬥機制改變了 AI 的實際行為，會破壞 Claude Code 的核心功能，用戶也無法信任輸出結果。

**決策**：戰鬥系統是純粹的 UI 層視覺化。Claude 正常處理任務，工具調用自動映射為魔法動畫。AI 的輸出完全不受戰鬥機制影響。

**效果**：用戶既能享受 RPG 戰鬥的樂趣，又能完全信任 AI 的實際工作成果。關閉遊戲化後，一切回到標準 CLI 體驗。

### 6.2 為什麼採用自動戰鬥而非手動操作？

**問題**：如果每次工具使用都需要用戶手動選擇「施放哪個魔法」，會大幅增加操作負擔，破壞 CLI 的高效體驗。

**決策**：Claude 的工具使用自動映射為攻擊。用戶只需正常輸入 Prompt，觀看戰鬥過程即可。夥伴也由 AI 自動控制。

**效果**：零額外操作負擔。用戶的日常開發流程完全不受影響，戰鬥只是增加了視覺趣味。

### 6.3 為什麼每次工具使用都是一次攻擊？

**問題**：需要找到一個自然的方式將 CLI 操作映射為戰鬥行為。

**決策**：每次工具調用（Read、Write、Edit、Bash、Grep、Glob）都是一次魔法攻擊。不同工具有不同屬性和威力，形成策略深度。

**效果**：
- 讓用戶感受到 Claude 在「戰鬥」— 每次文件讀寫都有視覺回饋
- 屬性相性系統讓不同工具在不同場景有不同效果
- 工具使用頻率越高，戰鬥越精彩

### 6.4 為什麼每場戰鬥自動建立 Worktree？

**問題**：任務型代碼修改直接在主分支進行有風險，用戶可能忘記手動建立分支。

**決策**：每場戰鬥自動建立獨立的 Worktree 分支。戰鬥結束後用戶選擇合併、保留或放棄。

**效果**：
- 每場戰鬥都有安全的隔離環境
- 戰鬥失敗或逃跑不會影響主分支
- 自然融入 RPG 敘事（每場戰鬥 = 一條平行世界時間線）
