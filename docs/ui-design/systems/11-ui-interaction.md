## 系統概述

### 核心設計原則

**1. Immediate Feedback（即時回饋）**：
- 所有操作都有即時的視覺/聽覺反饋
- HP/MP/EXP 變化顯示動畫過渡
- 技能施放有視覺效果和音效
- 按鈕點擊有按壓動畫

**2. State Visibility（狀態可見）**：
- HP/MP/EXP 始終在狀態欄可見
- 當前等級和資源顯著展示
- AI 處理狀態清晰指示
- 連接狀態始終顯示

**3. Reduced Cognitive Load（減少認知負擔）**：
- 預設技能降低使用門檻
- 常用任務通過快捷鍵訪問
- 從技能模板自動生成提示詞
- 清晰的視覺層次

**4. Progressive Disclosure（漸進式揭露）**：
- 高級功能初始隱藏
- 按需展開章節
- 避免壓倒新用戶
- 逐步引入功能

**5. Error Tolerance（錯誤容忍）**：
- 失敗有重試機制
- 自動保存對話
- 網絡重連處理
- 撤銷/取消操作

**6. Accessibility（無障礙）**：
- 鍵盤導航支持
- 屏幕閱讀器兼容
- 高對比模式
- 靈活字體大小

## 依賴關係

### 上游依賴

```
UI 互動系統依賴：
├─ 所有業務系統（L2/L3）- 提供數據和邏輯
├─ WebSocket（L1）- 實時通信
└─ 用戶配置（L2）- 個性化設置
```

### 下游依賴

```
影響的系統：
└─ 用戶體驗（最終用戶）
```

### 系統架構位置

```
L4: 展示層 ⭐
    └─ UI 互動系統（本系統）
L3: 業務層
    └─ 各種業務系統
L2: 核心層
    └─ 核心遊戲機制
L1: 基礎層
    └─ Bridge Layer
L0: 數據層
    └─ 數據存儲
```

## 核心規則

### 規則 1：界面布局要求

#### 1.1 狀態欄（頂部，固定）

**必須包含**：
- 玩家等級和名稱
- HP 條（當前/最大值）
- MP 條（當前/最大值）
- 金幣數量
- 通知圖標（帶徽章計數）
- 設置圖標
- 歷史圖標

**格式要求**：
```
❤️ HP: 80/100 [████████░░]
⚡ MP: 60/100 [██████░░░░]
💰 1,250
Lv.5 [玩家名稱]
```

#### 1.2 對話區域（可滾動）

**要求**：
- AI 頭像和名稱顯示
- 狀態指示器（思考中/施法中/完成）
- 消息氣泡帶時間戳
- 代碼塊語法高亮
- 每條消息的複製按鈕
- 重要消息的收藏按鈕
- 平滑滾動行為
- 虛擬滾動（性能優化）

#### 1.3 輸入區域

**要求**：
- 多行文本輸入支持
- 佔位符文本："告訴巫師你的需求..."
- 語音輸入按鈕
- 文件附件按鈕
- 發送按鈕（Enter 鍵）
- 清空對話按鈕
- 模式指示器（自由對話/技能模式）

**鍵盤快捷鍵**：
- Enter: 發送消息
- Shift+Enter: 新行
- Ctrl+K: 清空輸入
- Esc: 取消當前操作

#### 1.4 技能欄（底部或側邊）

**要求**：
- 顯示 5-6 個最常用技能
- 顯示技能圖標、名稱和 MP 消耗
- 指示技能狀態（可用/冷卻/鎖定/MP不足）
- 懸停顯示技能描述
- 點擊觸發技能參數對話框
- "顯示更多技能"按鈕
- 拖放重新排序支持

### 規則 2：技能施放要求

#### 2.1 驗證流程

```
用戶點擊技能
    ↓
檢查 MP 是否足夠
├─ 是 → 顯示參數對話框
└─ 否 → 顯示"MP 不足"錯誤
    ↓
用戶填寫參數
    ↓
生成完整提示詞
    ↓
顯示在輸入框
├─ 自動發送（可配置）
└─ 或等待用戶確認
    ↓
執行技能
├─ 播放施法動畫
├─ 扣除 MP（帶動畫）
└─ 發送到 AI
```

#### 2.2 參數對話框要求

**必須包含**：
- 技能名稱和圖標
- 參數輸入字段（帶標籤）
- MP 消耗顯示
- 取消按鈕
- 施法按鈕
- Tab 鍵在字段間導航
- Enter 鍵提交
- Esc 鍵取消

#### 2.3 動畫要求

**施法動畫序列**：
```
1. 技能圖標放大並發光（0.2s）
2. 魔法粒子效果（0.3s）
3. MP 條階梯動畫減少（0.3s）
4. 音效播放
5. AI 開始處理指示器顯示
```

---

### 規則 3：對話互動要求

#### 3.1 流式顯示

**要求**：
- 通過 WebSocket 接收塊
- 打字機效果顯示
- 實時渲染 Markdown
- 代碼塊語法高亮
- 打字時顯示光標動畫
- 速度：50ms/字符（可配置）

#### 3.2 消息格式支持

**必須支持**：
- 純文本
- **粗體** 和 *斜體*
- 內聯 `code`
- 代碼塊（帶語言規格）
- 列表（有序和無序）
- 鏈接（可點擊）
- 圖片（如適用）

#### 3.3 代碼塊要求

**要求**：
- 語言特定語法高亮
- 行號（可選）
- 懸停時出現複製按鈕
- 複製成功反饋（"✓ 已複製"，2秒）
- 保留正確縮進
- 長行換行（可配置）

---

### 規則 4：狀態變化要求

#### 4.1 HP/MP/EXP 更新

**動畫要求**：
- 進度條平滑填充/清空（0.3-0.5s）
- 浮動數字顯示變化量
- 顏色指示類型（綠色=增加，紅色=減少）
- 每種變化類型的音效

**MP 消耗**：
```
之前：⚡ ██████████ 60/100
之後：⚡ █████░░░░░ 50/100
      ↑ 階梯動畫，從右到左
      ↑ 音效："mp_down.wav"
```

**HP 恢復**：
```
之前：❤️ ████░░░░░░ 40/100
之後：❤️ ██████░░░░ 60/100
      ↑ 填充動畫，從左到右
      ↑ 音效："heal.wav"
      ↑ 浮動文本："+20 HP"
```

#### 4.2 升級要求

**觸發**：當 EXP 達到下一級所需量時

**動畫序列**：
```
1. EXP 條填充到 100%
2. 閃光效果（0.3s）
3. 等級數字帶縮放動畫變化（0.5s）
4. 全屏慶祝模態框出現
5. 顯示獎勵（HP/MP 增加，新技能，金幣）
6. 音效："level_up.wav"
7. 允許用戶關閉
```

**模態框必須顯示**：
- 舊等級 → 新等級
- HP 最大值增加
- MP 最大值增加
- 金幣獎勵
- 新解鎖技能（如有）
- "太棒了！"按鈕關閉

### 規則 5：特殊場景要求

#### 5.1 連接失敗

**要求**：
- 清晰的錯誤消息
- 故障排除步驟
- 重試按鈕
- 設置指南鏈接
- 每 5 秒自動重試（最多 10 次）

**錯誤消息必須包含**：
- 出了什麼問題
- 可能的原因
- 解決步驟
- 支持鏈接

#### 5.2 MP 耗盡

**要求**：
- 阻止新請求
- 顯示"巫師需要休息"模態框
- 顯示當前 MP 和最大 MP
- 顯示恢復選項：
  - 等待自動恢復（顯示倒計時）
  - 完成每日任務
  - 使用 MP 恢復道具（如可用）

**恢復速率**：
- 自動恢復：20 MP/小時
- 每日任務：30-50 MP/任務
- MP 藥水：50 MP 即時

#### 5.3 首次用戶引導

**要求**：
- 3 步引導教程
- 步驟 1：輸入玩家名稱
- 步驟 2：選擇角色職業
- 步驟 3：嘗試第一個技能
- 任何步驟可跳過
- 完成後永不再顯示

### 規則 6：無障礙要求

#### 6.1 鍵盤導航

**Tab 順序**：
```
1. 輸入框
2. 發送按鈕
3. 技能按鈕（從左到右）
4. 頂欄按鈕（通知、設置、歷史）
5. 對話中的互動元素
```

**鍵盤快捷鍵**：
- Tab: 下一個元素
- Shift+Tab: 上一個元素
- Enter: 發送消息/激活按鈕
- Esc: 關閉模態框/取消操作
- Ctrl+H: 打開歷史
- Ctrl+,: 打開設置
- Ctrl+1~5: 使用技能 1-5

#### 6.2 屏幕閱讀器支持

**要求**：
- 所有互動元素有 aria-labels
- 狀態變化通過 aria-live 宣布
- 進度條有文本等效
- 按鈕描述其動作
- 表單有適當標籤

**示例**：
```html
<button aria-label="代碼生成技能，消耗 10 MP">
  ⚔️ 代碼生成
</button>
```

#### 6.3 高對比模式

**要求**：
- 檢測系統偏好 `prefers-contrast: high`
- 邊框寬度增加到 3px
- 使用純黑白
- 保持 7:1 對比度
- 避免僅依賴顏色

---

### 規則 7：性能要求

#### 7.1 響應時間目標

```
performanceTargets = {
  initialLoad: 3000,     // 初始加載 < 3 秒
  skillActivation: 100,  // 技能激活 < 100ms
  messageSend: 200,      // 消息發送 < 200ms
  firstAIChunk: 2000,    // 首個 AI 響應塊 < 2 秒
  animationFps: 60       // UI 動畫幀率 60 FPS
}
```

#### 7.2 優化要求

**虛擬滾動**：
- 僅渲染可見消息（± 5 緩衝）
- 卸載屏幕外內容
- 更新時保持滾動位置

**懶加載**：
- 按需加載圖片
- 延遲非關鍵腳本
- 漸進增強

**資源管理**：
- 存儲消息限制為 1000 條
- 自動歸檔舊對話
- 低內存時清除緩存

---

### 規則 8：錯誤處理要求

#### 8.1 錯誤消息標準

**不好的示例**：
```
Error: ENOENT: no such file or directory
```

**好的示例**：
```
┌──────────────────────────────┐
│  ⚠️  找不到 claude-cli      │
│                              │
│  請安裝:                     │
│  $ npm install -g @anthropi…│
│                              │
│  [設置指南] [重試]           │
└──────────────────────────────┘
```

**要求**：
- 使用友好語言
- 解釋出了什麼問題
- 提供可操作步驟
- 提供幫助/文檔鏈接
- 包含重試機制

#### 8.2 網絡錯誤處理

**要求**：
- 立即檢測斷線
- 顯示重連嘗試進度
- 保存未發送消息
- 重連時自動恢復
- 通知用戶成功/失敗

## 系統整合

### 與所有業務系統整合

UI 互動系統是最頂層，與所有業務系統整合：

```
uiSystem.integrate({
  battle: battleSystem,
  scene: sceneSystem,
  companion: companionSystem,
  skill: skillSystem,
  shop: shopSystem,
  // ... 所有其他系統
});

// 監聽事件並更新 UI
battleSystem.on('hp_change', (change) => {
  uiSystem.animateHpChange(change);
});

skillSystem.on('skill_used', (skill) => {
  uiSystem.showSkillAnimation(skill);
});
```

## 設計決策

### 決策 1：為什麼需要即時反饋？

**理由**：
- 用戶需要知道操作是否成功
- 即時反饋增加信心和滿意度
- 動畫讓界面更生動
- 音效增強沉浸感

### 決策 2：為什麼需要無障礙支持？

**理由**：
- 所有用戶都應能使用
- 法律要求（某些地區）
- 鍵盤導航提高效率
- 屏幕閱讀器支持視障用戶

### 決策 3：為什麼需要性能優化？

**理由**：
- 流暢體驗提高滿意度
- 低端設備也能運行
- 長對話不卡頓
- 節省用戶資源

### 決策 4：為什麼需要錯誤容忍？

**理由**：
- 用戶會犯錯
- 網絡不穩定
- AI 可能失敗
- 提供重試機會

### 決策 5：為什麼需要漸進式揭露？

**理由**：
- 避免壓倒新用戶
- 逐步引入功能
- 高級用戶可展開
- 降低認知負擔
