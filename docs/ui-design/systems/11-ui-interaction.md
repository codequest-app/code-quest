# L4 UI 互動系統

**版本**: v2.0
**更新日期**: 2026-02-06

---

## 一、系統概述

UI 互動系統是 Code Quest 的**表現層**，定義所有使用者互動模式、視覺回饋、無障礙標準和效能目標。它是所有其他系統面向使用者的唯一出口 — 無論底層系統如何運作，最終都透過 UI 互動系統呈現給玩家。

在系統架構中，UI 互動系統位於 **L4 表現層**，是架構最頂層的系統。它依賴所有底層系統的資料與狀態，將其轉化為可見、可互動、可理解的介面元素。從狀態列的 HP/MP 顯示，到技能施放動畫，到戰鬥進度回饋，一切視覺體驗都由本系統定義。

## 三、核心規則

### 3.1 六大設計原則

| 編號 | 原則 | 說明 | 實踐 |
|------|------|------|------|
| 1 | 即時回饋 | 所有操作必須有即時的視覺或聽覺回應 | 點擊技能立即播放動畫，不等待網路回應 |
| 2 | 狀態可見 | HP/MP/EXP 等核心數值始終可見 | 固定在頂部的狀態列，任何頁面都不隱藏 |
| 3 | 降低認知負擔 | 預設值與快捷鍵降低使用門檻 | 預設推薦模式、常用技能一鍵施放 |
| 4 | 漸進揭露 | 進階功能初始隱藏，隨等級解鎖 | 低等級只看到基礎技能，高等級解鎖進階面板 |
| 5 | 容錯設計 | 重試、自動儲存、斷線重連 | 操作失敗不丟失輸入，自動嘗試恢復 |
| 6 | 無障礙 | 鍵盤導航、螢幕閱讀器、高對比 | 符合 WCAG AA 標準 |

### 3.2 介面區域劃分

介面分為四個主要區域，各有明確職責：

```
狀態列（固定頂部）
  ├─ 等級顯示
  ├─ HP 血條（紅色）
  ├─ MP 魔力條（藍色）
  ├─ 金幣數量
  ├─ 當前位置
  └─ 通知指示器

對話區域（可捲動中央）
  ├─ AI 回應訊息（串流顯示、打字機效果）
  ├─ 使用者訊息
  ├─ 代碼區塊（語法高亮）
  ├─ 戰鬥結果摘要
  └─ 系統通知

輸入區域（固定底部）
  ├─ 多行輸入框
  ├─ 傳送按鈕
  ├─ 當前模式指示器（探索/戰鬥/規劃）
  └─ AI 模型指示器

技能列（底部或側邊）
  ├─ 5-6 個快速存取技能位
  ├─ 每個位顯示：圖示、名稱、MP 消耗、冷卻狀態
  └─ 技能狀態指示：可用（亮）/ 冷卻中（灰）/ MP 不足（暗紅）
```

### 3.3 技能施放流程

從點擊技能到發送 AI 請求的完整流程：

```
步驟 1 — 點擊技能
  使用者點擊技能列中的技能圖示
  或使用快捷鍵 Ctrl+1 ~ Ctrl+5

步驟 2 — MP 檢查
  系統檢查當前 MP 是否足夠
  不足時：技能圖示閃紅 + 提示 "MP 不足"，流程終止

步驟 3 — 參數對話框
  部分技能需要額外參數（如目標檔案、範圍）
  顯示參數輸入對話框，預設值自動填入

步驟 4 — 生成 Prompt
  將技能模板 + 使用者參數組合為完整 Prompt

步驟 5 — 施放動畫
  播放技能施放動畫（約 1.1 秒）
  動畫期間扣除 MP，MP 條平滑減少

步驟 6 — 發送 AI
  將 Prompt 發送至 AI（經由多模型路由）
  對話區域顯示串流回應
```

### 3.4 效能目標

| 指標 | 目標值 | 說明 |
|------|-------|------|
| 初始載入 | < 3 秒 | 首次開啟應用到可互動的時間 |
| 技能啟動 | < 100ms | 點擊技能到開始播放動畫 |
| 訊息發送 | < 200ms | 按下傳送到請求離開客戶端 |
| 首次 AI 回應 | < 2 秒 | 發送到開始顯示串流文字 |
| 動畫幀率 | 60 FPS | 所有動畫維持流暢 |
| 訊息列表 | 虛擬捲動 | 只渲染可見區域的訊息 |
| 訊息上限 | 1000 則 | 超過自動歸檔，保持效能 |

### 3.5 無障礙標準（WCAG AA）

#### 鍵盤導航

```
全域快捷鍵：
  Tab          — 在介面區域間移動焦點
  Ctrl+1 ~ 5  — 快速施放技能列第 1-5 個技能
  Ctrl+Enter  — 發送訊息
  Escape      — 關閉對話框 / 取消操作
  Ctrl+/      — 開啟快捷鍵說明

焦點順序：
  狀態列 → 對話區域 → 輸入區域 → 技能列
  對話框開啟時焦點鎖定在對話框內（焦點陷阱）
```

#### 螢幕閱讀器

```
互動元素：所有按鈕、輸入框、技能圖示附加 aria-label
動態內容：狀態變更使用 aria-live 即時通報
  ├─ HP/MP 變化：aria-live="polite"
  ├─ 戰鬥開始/結束：aria-live="assertive"
  └─ AI 回應串流：aria-live="polite"
結構標記：區域使用 landmark roles（banner、main、contentinfo）
```

#### 高對比模式

```
偵測：prefers-contrast: high 媒體查詢
最低對比度：7:1（WCAG AAA 等級）
邊框寬度：最低 3px（確保元素邊界清晰）
焦點指示：3px 實線外框 + 2px 偏移
色彩不作為唯一資訊載體（搭配圖示或文字）
```

#### 彈性字體

```
基礎字號支持 100%-200% 縮放
不使用固定像素值限制最大字號
行高隨字號等比放大
容器高度自適應內容
```

### 3.6 視覺風格

**Pixel Art（8-bit / 16-bit 風格）**

```
字體配置：
  ├─ 標題字體：Press Start 2P（像素風格裝飾字體）
  ├─ 內文字體：VT323（像素風格等寬內文字體）
  └─ UI 字體：Roboto Mono（程式碼與介面數值）

風格要素：
  ├─ 像素化圖示與角色立繪
  ├─ 8-bit 色彩調色盤
  ├─ 復古邊框與對話框樣式
  └─ 點陣風格的進度條與血條
```

### 3.7 狀態變化動畫

| 變化類型 | 動畫描述 | 時長 |
|---------|---------|------|
| 進度條填充 | 平滑填充至目標值 | 0.3-0.5 秒 |
| 數值變化 | 浮動數字顯示差值（+50 HP、-30 MP） | 1.0 秒後消失 |
| 增益效果 | 綠色數字向上浮動 | 1.0 秒 |
| 損失效果 | 紅色數字向上浮動 | 1.0 秒 |
| EXP 獲得 | 經驗條先快後慢填充 | 0.5-1.0 秒 |
| 金幣獲得 | 金幣圖示跳動 + 數字滾動 | 0.5 秒 |

### 3.8 錯誤處理

所有錯誤訊息遵循「說明 + 原因 + 方案」三段式結構：

```
錯誤訊息格式：
  ┌────────────────────────────┐
  │  ⚠ 連線中斷                 │
  │                            │
  │  與 AI 服務的連線已斷開。    │
  │  可能是網路不穩定所致。      │
  │                            │
  │  [重試連線]  [離線模式]     │
  └────────────────────────────┘

自動重連機制：
  間隔：每 5 秒嘗試一次
  上限：最多 10 次
  超過上限：顯示手動重連按鈕
  重連期間：介面可用，僅 AI 功能暫停
```

### 3.9 升級動畫序列

當冒險者獲得足夠經驗值升級時，觸發完整的慶祝序列：

```
階段 1 — 經驗條填滿
  EXP 條加速填充至 100%
  填滿瞬間閃白光

階段 2 — 等級數字動畫
  等級數字放大 → 縮回 → 彈跳效果
  舊等級淡出，新等級淡入

階段 3 — 全畫面慶祝
  短暫的全畫面特效（粒子、光芒）
  持續約 1.5 秒

階段 4 — 獎勵展示
  顯示本次升級獲得的獎勵：
    新技能解鎖
    屬性提升
    新區域開放

階段 5 — 關閉
  使用者點擊任意處關閉
  或 5 秒後自動消失
```

## 五、與其他系統的互動

### 5.1 UI ← 角色系統 (4) — 狀態列

狀態列持續顯示角色系統的核心數值：

```
角色系統提供的資料：
  ├─ 等級（Level）→ 狀態列左側顯示
  ├─ HP / MaxHP  → 紅色血條（百分比填充）
  ├─ MP / MaxMP  → 藍色魔力條（百分比填充）
  ├─ EXP / NextLevelEXP → 經驗條（狀態列下方細條）
  └─ Gold        → 金幣數字

更新觸發：角色系統數值變動時即時通知 UI 更新
動畫處理：數值變化時播放平滑過渡 + 浮動數字
```

### 5.2 UI ← 技能系統 (5) — 技能列

技能列顯示技能系統的可用技能：

```
技能系統提供的資料：
  ├─ 已裝備技能列表（最多 5-6 個）
  ├─ 每個技能的 MP 消耗
  ├─ 冷卻狀態（剩餘回合數）
  └─ 技能等級與描述

UI 渲染：
  ├─ 可用技能：正常亮度，可點擊
  ├─ 冷卻中：灰色覆蓋 + 倒數數字
  ├─ MP 不足：暗紅色調 + 鎖定圖示
  └─ 未解鎖：問號圖示 + 等級需求提示
```

### 5.3 UI ← 戰鬥系統 (3) — 戰鬥介面

戰鬥開始時 UI 切換至戰鬥模式：

```
戰鬥系統提供的資料：
  ├─ 怪物資訊（名稱、等級、HP）
  ├─ 回合資訊（當前回合、誰的行動）
  ├─ 傷害計算結果（用於浮動數字顯示）
  └─ 戰鬥結果（勝負、獎勵）

UI 介面切換：
  ├─ 對話區域增加戰鬥元素（怪物圖示、血條）
  ├─ 傷害數字浮動顯示
  ├─ 回合指示器
  └─ 戰鬥結束：結算面板（EXP、金幣、掉落物）
```

### 5.4 UI ← 非同步戰鬥系統 (9) — 三面板

## 六、設計決策說明

### 6.1 為什麼狀態列始終可見？

**問題**：如果 HP/MP/EXP 只在特定頁面可見，使用者容易忘記當前狀態，可能在低 HP 時冒然進入危險區域，或在 MP 不足時嘗試施放技能。

**決策**：狀態列固定在頂部，無論在任何頁面、任何模式下都可見。核心數值（等級、HP、MP、金幣）永遠顯示。

**效果**：使用者始終掌握角色狀態，做出更明智的決策。減少「驚喜失敗」的挫敗感。

### 6.2 為什麼選擇 Pixel Art 風格？

**問題**：視覺風格需要與 RPG 主題一致，同時考慮資產製作成本和技術可行性。

**決策**：採用 8-bit / 16-bit Pixel Art 風格。搭配 Press Start 2P（標題）、VT323（內文）、Roboto Mono（UI）三種字體。

**效果**：
- 復古趣味氛圍 — 與 RPG 主題完美契合
- 資產製作簡單 — 像素圖比精細插畫製作門檻低
- 風格統一 — 所有元素天然視覺一致
- 檔案體積小 — 像素圖壓縮率高，利於效能

### 6.3 為什麼技能列提供快捷鍵？

**問題**：開發者習慣鍵盤操作，頻繁使用滑鼠點擊技能會打斷工作流程。

**決策**：技能列支援 Ctrl+1 到 Ctrl+5 快捷鍵，一鍵施放對應位置的技能。

**效果**：熟練使用者可以完全不離開鍵盤完成技能施放。搭配多行輸入框的 Ctrl+Enter 傳送，實現全鍵盤操作流程。

### 6.4 為什麼使用虛擬捲動？

**問題**：長時間使用後，對話區域可能累積數百甚至數千則訊息。如果全部渲染在 DOM 中，會導致嚴重的效能問題。

**決策**：對話區域使用虛擬捲動（Virtual Scrolling），只渲染當前可見區域的訊息。設定 1000 則上限，超過自動歸檔。

**效果**：無論累積多少訊息，渲染效能恆定。1000 則上限足以覆蓋一整天的使用量，歸檔訊息仍可查閱。

### 6.5 為什麼無障礙從一開始就納入？

**問題**：無障礙功能如果事後追加，改造成本極高，且容易遺漏。許多互動模式需要從架構層面支援。

**決策**：六大設計原則中將無障礙列為基礎原則，所有元素從設計階段就考慮鍵盤導航、螢幕閱讀器、高對比支援。

**效果**：符合 WCAG AA 標準，覆蓋更廣泛的使用者群體。避免事後改造的巨大成本。鍵盤導航同時也提升了一般使用者的操作效率。

### 6.6 為什麼錯誤訊息採用三段式結構？

**問題**：技術性錯誤訊息（如 "Error 500: Internal Server Error"）對使用者毫無幫助，使用者不知道發生了什麼、為什麼、該怎麼辦。

**決策**：所有錯誤訊息統一採用「說明 + 原因 + 方案」三段式。說明用白話解釋發生了什麼，原因解釋可能的原因，方案提供具體的操作按鈕。

**效果**：使用者面對錯誤時不焦慮，知道發生了什麼且有明確的下一步行動。自動重連機制進一步減少使用者手動介入的需要。
