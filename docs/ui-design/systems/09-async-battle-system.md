# L3 非同步戰鬥系統

**版本**: v2.0
**更新日期**: 2026-02-06

---

## 一、系統概述

非同步戰鬥系統是 Code Quest 的並行任務處理核心，採用**非阻塞雙軌架構**。當冒險者發出複雜任務時，系統可在背景執行戰鬥（代碼任務），同時讓冒險者繼續在前台進行對話或發起新的戰鬥。每個並行戰鬥會自動建立獨立的 Worktree，確保代碼隔離安全。

在系統架構中，非同步戰鬥系統位於 **L3 進階層**，是整個系統中唯一處理「並行執行」的系統。它將使用者輸入智慧分流為三條執行路徑，讓簡單問答即時回應、中等任務同步執行、複雜任務背景處理，實現「使用者輸入永不阻塞」的核心目標。

---

## 二、依賴關係

```
依賴：
  ├─ 戰鬥系統 (3)     — 戰鬥流程、回合機制、傷害計算
  └─ Worktree 系統 (8) — 自動建立隔離工作樹進行並行任務

被依賴：無（終端系統）
```

| 方向 | 系統 | 關係說明 |
|------|------|---------|
| 依賴 | 戰鬥系統 (3) | 非同步戰鬥的每個實例都是一場完整戰鬥，遵循戰鬥系統的流程與規則 |
| 依賴 | Worktree 系統 (8) | Battle Async 路徑自動呼叫 Worktree 系統建立隔離工作目錄 |

## 三、核心規則

### 3.1 SmartRouter 智慧路由

所有使用者輸入首先經過 SmartRouter 進行複雜度分析，根據分數分流至三條執行路徑：

| 路徑 | 分數範圍 | 執行方式 | 耗時 | 戰鬥 UI | Worktree |
|------|---------|---------|------|---------|----------|
| 對話軌道（Dialog Track） | 0-2 | 主執行緒，立即回應 | <1s | 無 | 無 |
| 主線同步（Main Sync） | 3-7 | 主執行緒，同步執行 | 5s-1min | 有（等待） | 無 |
| 戰鬥非同步（Battle Async） | 8+ | 子程序，立即返回 | 1min-數小時 | 有（背景） | 自動建立 |

**路徑特性**：

```
對話軌道 (Dialog Track)
  分數：0-2
  特性：主執行緒、即時、不產生戰鬥 UI
  範例："什麼是遞迴？"、"顯示狀態"、"解釋這段程式碼"
  本質：純粹的 AI 對話，不涉及代碼修改

主線同步 (Main Sync)
  分數：3-7
  特性：主執行緒同步執行，顯示戰鬥 UI 但使用者須等待
  範例："修復型別錯誤"、"新增一個函式"、"更新 README"
  本質：單一執行緒 = 安全，不需要 Worktree 隔離

戰鬥非同步 (Battle Async)
  分數：8+
  特性：子程序非同步執行，立即返回控制權，自動建立 Worktree
  範例："重構認證系統"、"遷移架構"、"實作完整測試覆蓋"
  本質：需要隔離環境，長時間執行不阻塞使用者
```

### 3.2 複雜度評分機制

SmartRouter 透過 5 個維度計算複雜度分數，滿分 15 分：

| 維度 | 分數範圍 | 評分標準 |
|------|---------|---------|
| 輸入長度 | 0-3 | >200 字元 = +3，100-200 = +2，<100 = +1 |
| 關鍵字權重 | 0-3 | 重量級（×3）/ 中量級（×2）/ 輕量級（×1） |
| 檔案數量 | 0-3 | 多檔案 = +3，隱含多檔 = +2，單一檔案 = +1 |
| 工具複雜度 | 0-3 | 多工具 = +3，複雜工具（Plan）= +2，簡單工具 = +1 |
| 相依性 | 0-2 | 系統間相依 = +2，獨立任務 = +0 |

**關鍵字分級**：

```
重量級關鍵字（×3 倍權重）：
  refactor / migrate / entire / system / architecture / overhaul

中量級關鍵字（×2 倍權重）：
  create / implement / optimize / redesign / integrate

輕量級關鍵字（×1 倍權重）：
  fix / update / check / add / modify / review
```

**路由判定**：

```
總分計算 = 長度分 + 關鍵字分 + 檔案分 + 工具分 + 相依分

  總分 < 3   → 對話軌道（Dialog Track）
  總分 3-7   → 主線同步（Main Sync）
  總分 ≥ 8   → 戰鬥非同步（Battle Async）
```

### 3.3 戰鬥實例池（Battle Instance Pool）

| 規則 | 數值 | 說明 |
|------|------|------|
| 最大並行數 | 3 | 同時最多 3 場非同步戰鬥 |
| 溢出處理 | 排隊 | 第 4 場以上進入等待佇列 |
| 超時機制 | 30 分鐘 | 超時自動終止並回報 |
| 實例命名 | `battle_<id>` | 與手動 Worktree 區分（手動使用使用者命名） |

### 3.4 Battle Async Worktree 生命週期

戰鬥非同步路徑的 Worktree 從建立到清理的完整流程：

```
階段 1 — 自動建立
  指令：git worktree add worktrees/battle_<id> main
  時機：Battle Async 路由確定後立即執行
  命名：battle_<id>（與手動 Worktree 使用者命名區分）

階段 2 — 隔離執行
  在獨立目錄中執行 Claude CLI
  所有檔案修改限定在 Worktree 目錄內
  不影響主工作目錄

階段 3 — 完成合併
  戰鬥完成後自動合併回主分支
  衝突時暫停並通知使用者處理

階段 4 — 清理移除
  合併成功後自動移除 Worktree
  git worktree remove worktrees/battle_<id>
```

## 五、與其他系統的互動

### 5.1 非同步戰鬥 ← 戰鬥系統 (3)

每場非同步戰鬥都是一個完整的戰鬥實例，遵循戰鬥系統的所有規則：

```
戰鬥系統提供：
  ├─ 戰鬥流程（開始 → 回合 → 結束）
  ├─ 傷害計算公式
  ├─ 經驗值 / 金幣獎勵
  └─ 怪物生成邏輯

非同步戰鬥擴展：
  ├─ 將同步流程包裝為非同步子程序
  ├─ 增加進度回報機制
  ├─ 增加超時 / 取消控制
  └─ 增加並行管理（實例池）
```

### 5.2 非同步戰鬥 ← Worktree 系統 (8)

Battle Async 路徑自動呼叫 Worktree 系統建立隔離環境：

```
手動 Worktree（使用者在公會大廳操作）：
  命名：使用者自訂名稱（如 feature-auth）
  管理：使用者手動建立 / 切換 / 合併 / 刪除
  場所：僅限城鎮公會大廳

戰鬥 Worktree（系統自動管理）：
  命名：battle_<id>（如 battle_001）
  管理：系統自動建立、完成後自動合併、自動清理
  場所：不受地圖限制，任何位置均可觸發
```

- 兩種 Worktree 透過命名前綴區分，互不干擾
- 戰鬥 Worktree 在 Worktree 列表中以特殊標記顯示
- 使用者無法手動操作戰鬥 Worktree（防止意外破壞進行中的戰鬥）

### 5.3 非同步戰鬥 → 多模型整合系統 (10)

SmartRouter 完成分流後，Battle Async 路徑會進一步呼叫 MultiModelRouter 選擇最佳模型：

```
SmartRouter（分數 ≥ 8）
    │
    ▼
MultiModelRouter
    ├─ 預設平衡模式：複雜度 8+ → Claude（深度推理）
    ├─ 成本優先模式：優先 Gemini（節省成本）
    └─ 每場戰鬥獨立追蹤模型用量與成本
```

## 六、設計決策說明

### 6.1 為什麼使用者輸入永不阻塞？

**問題**：傳統 CLI 工具在執行長時間任務時，使用者必須等待完成才能輸入下一個指令。重構一個模組可能需要數分鐘，這段時間使用者完全閒置。

**決策**：採用非阻塞雙軌架構，所有路徑都立即回應使用者。Dialog Track 直接回應、Main Sync 顯示進度但佔用主執行緒、Battle Async 立即返回控制權。

**效果**：使用者可以在等待重構完成的同時，詢問技術問題、查看狀態、甚至發起另一場戰鬥。大幅提升工具的實用性與使用者體驗。

### 6.2 為什麼需要 SmartRouter 而非讓使用者自己選擇？

**問題**：如果讓使用者手動選擇「同步」或「非同步」執行，增加了認知負擔，且多數使用者無法準確判斷任務複雜度。

**決策**：SmartRouter 透過 5 維度自動評分，智慧分流至對應路徑。使用者只需輸入指令，系統自動決定最佳執行方式。

**效果**：簡單問答不會產生不必要的 Worktree 開銷，複雜任務自動獲得隔離環境。使用者體驗流暢無感。

### 6.3 為什麼最大並行數限制為 3？

**問題**：無限制的並行戰鬥會耗盡系統資源（CPU、記憶體、API 配額），且每個 Worktree 是完整的程式碼副本，佔用磁碟空間。

**決策**：硬性限制最大 3 個並行戰鬥，第 4 個以上進入排隊。搭配 30 分鐘超時機制防止殭屍程序。

**效果**：3 個並行已足以覆蓋大多數使用場景（如同時重構、測試、文件撰寫），同時保持系統穩定。佇列機制確保不會丟失任務。

### 6.4 為什麼 Main Sync 不需要 Worktree？

**問題**：為所有戰鬥都建立 Worktree 會增加不必要的開銷（建立和合併的時間成本）。

**決策**：Main Sync（分數 3-7）在主執行緒同步執行，因為單一執行緒不存在並行寫入衝突，無需隔離。只有 Battle Async（分數 8+）才自動建立 Worktree。

**效果**：中等複雜度的任務（如修復 bug、新增函式）快速完成，無需等待 Worktree 建立和合併流程。只有真正需要長時間執行的任務才獲得隔離保護。

### 6.5 為什麼戰鬥 Worktree 與手動 Worktree 分離？

**問題**：如果戰鬥自動建立的 Worktree 與使用者手動管理的混在一起，使用者可能誤刪正在執行的戰鬥環境，或戰鬥清理時誤刪使用者的工作分支。

**決策**：透過命名規則嚴格區分 — 戰鬥使用 `battle_<id>` 前綴，手動使用使用者自訂名稱。戰鬥 Worktree 由系統全自動管理，使用者不可操作。

**效果**：兩套 Worktree 各自獨立運作，互不干擾。使用者在公會大廳看到的列表清楚區分「我的分支」與「進行中的戰鬥」。
