# L0 場景系統

**版本**: v2.0
**更新日期**: 2026-02-06

---

## 一、系統概述

場景系統是 Code Quest 的雙場景核心，負責區分**探索模式**與**戰鬥模式**。兩種模式擁有完全不同的 UI 佈局、操作規則、和資源管理策略。

在系統架構中，場景系統位於 **L0 基礎層**，依賴地圖系統提供的空間位置，為上層的戰鬥、商店、Worktree、互動事件等系統提供場景狀態。它是連接「空間」與「行為」的橋樑 — 地圖系統決定玩家在哪裡，場景系統決定玩家能做什麼。

---

## 二、依賴關係

```
依賴：
  └─ 地圖系統 (1) — 需要知道玩家所在區域來決定場景規則

被依賴：
  ├─ 戰鬥系統 (3)    — 戰鬥只在戰鬥模式中進行
  ├─ 商店系統 (6)    — 商店只在探索模式中可用
  ├─ Worktree 系統 (8) — Worktree 操作只在探索模式中可用
  └─ 互動事件系統 (7) — Plan Mode 需要場景暫停機制
```

| 方向 | 系統 | 關係說明 |
|------|------|---------|
| 依賴 | 地圖系統 (1) | 根據玩家所在區域（城鎮/野外/副本）決定可觸發的場景 |
| 被依賴 | 戰鬥系統 (3) | 場景切換到戰鬥模式時，啟動戰鬥系統 |
| 被依賴 | 商店系統 (6) | 只有在探索模式下才允許訪問商店 |
| 被依賴 | Worktree 系統 (8) | 只有在探索模式下才允許操作 Worktree |
| 被依賴 | 互動事件系統 (7) | Plan Mode 等互動事件需要暫停當前場景 |

## 三、核心規則

### 3.1 雙場景模型

遊戲在任何時刻只會處於兩種場景之一：

| 屬性 | 探索模式 | 戰鬥模式 |
|------|---------|---------|
| 目的 | 管理、休息、對話、規劃 | 完成任務、獲得獎勵 |
| 觸發條件 | 應用啟動、戰鬥結束 | 任務型 Prompt + 複雜度 ≥ 8 |
| 可用操作 | 主選單、對話、商店、Worktree | 技能施放、夥伴指揮、召喚、逃跑 |
| HP 恢復速率 | +2 HP/秒 | 0 HP/秒（不恢復） |
| MP 恢復速率 | +1 MP/秒 | +0.1 MP/秒（極慢） |
| Worktree 操作 | 允許 | 鎖定（自動建立的戰鬥分支除外） |

### 3.2 探索模式詳述

探索模式是預設場景，提供以下功能：

**主選單項目**：
1. 狀態（Status） — 查看 HP/MP/EXP/等級
2. 技能（Skills） — 查看已解鎖的技能列表
3. 夥伴（Companions） — 查看夥伴狀態與配置
4. 召喚獸（Summons） — 查看召喚獸圖鑑
5. 成就（Achievements） — 查看解鎖的成就
6. Worktree — 平行世界管理
7. 設定（Settings） — 遊戲設定

**快速資源恢復**：
- HP：每秒 +2，鼓勵戰鬥間回城休整
- MP：每秒 +1，確保下次戰鬥有足夠魔力

### 3.3 戰鬥模式詳述

戰鬥模式在遭遇觸發後進入，擁有獨立的操作集：

**可用操作**：
- 技能施放（Claude 工具使用自動映射）
- 夥伴指揮（召喚/切換/指令）
- 召喚獸使用（爆發支援）
- 逃跑（消耗 30% 當前 MP）

**緩慢資源恢復**：
- HP：每秒 0，戰鬥中不恢復生命
- MP：每秒 +0.1，僅提供極少量魔力回復

**Worktree 鎖定**：
- 戰鬥開始時自動建立戰鬥分支
- 戰鬥中無法手動操作 Worktree
- 戰鬥結束後提示合併/保留/放棄

### 3.4 場景切換規則

```
探索 ──→ 戰鬥：任務型 Prompt + 複雜度 ≥ 8 + 位於野外或副本
戰鬥 ──→ 探索：勝利（敵人 HP = 0）或逃跑（消耗 30% MP）

特殊切換：
  任意場景 ──→ 靜止之間：Plan Mode 觸發 / 玩家主動進入
  靜止之間 ──→ 返回原場景：計劃完成 / 玩家退出
```

**逃跑規則**：
- 消耗當前 MP 的 30%
- 不獲得任何獎勵（EXP、Gold 均為 0）
- Worktree 戰鬥分支保留（可稍後在公會大廳管理）
- 返回探索模式

### 3.5 Prompt 分析器

場景系統內建 Prompt 分析器，負責判斷用戶輸入屬於對話型還是任務型。

**任務型關鍵字**（觸發遭遇判定）：

| 類別 | 關鍵字 |
|------|--------|
| 動作詞 | 修復、創建、實作、重構、優化、測試、部署、配置 |
| 代碼詞 | bug、功能、api、組件、函數、類別、模組 |
| 英文詞 | fix, create, implement, refactor, optimize, test, deploy |

**對話型關鍵字**（不觸發遭遇）：

| 類別 | 關鍵字 |
|------|--------|
| 問題詞 | 什麼、為什麼、怎麼、如何、是否 |
| 解釋詞 | 解釋、說明、介紹、比較、區別 |
| 英文詞 | what, why, how, explain, describe, compare |

### 3.6 複雜度評分

分析 Prompt 的複雜度，決定是否達到戰鬥觸發門檻（≥ 8）：

| 評分因素 | 說明 | 影響方向 |
|---------|------|---------|
| Prompt 長度 | 字數越多 | 複雜度越高 |
| 動作關鍵字數量 | 多個動作詞 | 複雜度越高 |
| 多步驟指令 | 包含「並且」「然後」等連接詞 | 複雜度越高 |
| 技術深度 | 涉及高級技術詞彙 | 複雜度越高 |

**評分範例**：

```
「修復按鈕 Bug」          → 複雜度 3  → 不觸發戰鬥（主 CLI 直接處理）
「實作用戶登入功能」       → 複雜度 10 → 觸發戰鬥
「重構認證系統並確保向後兼容」→ 複雜度 15 → 觸發強敵
```

## 五、與其他系統的互動

### 5.1 場景 ← 地圖系統 (1)

場景系統從地圖系統獲取玩家位置，用於：

- **城鎮區域**：強制探索模式，Prompt 分析器跳過遭遇判定
- **野外區域**：啟用完整的 Prompt 分析流程
- **副本區域**：強制進入戰鬥模式，跳過 Prompt 分析

### 5.2 場景 → 戰鬥系統 (3)

當場景切換到戰鬥模式時：

```
場景系統                        戰鬥系統
   │                               │
   │  觸發戰鬥（敵人資訊、區域）    │
   │──────────────────────────────►│
   │                               │  初始化戰鬥實例
   │                               │  生成敵人
   │                               │  開始回合制
   │  戰鬥結果（勝利/逃跑）        │
   │◄──────────────────────────────│
   │                               │
   │  切換回探索模式                │
```

### 5.3 場景 → 商店系統 (6)

- 場景為探索模式時：商店系統可正常訪問
- 場景為戰鬥模式時：商店系統入口鎖定

### 5.4 場景 → Worktree 系統 (8)

- 場景為探索模式時：允許手動 Worktree 操作（建立/切換/合併）
- 場景為戰鬥模式時：Worktree 操作鎖定（僅自動建立的戰鬥分支例外）

### 5.5 場景 → 互動事件系統 (7)

當互動事件（Plan Mode、AskUserQuestion、錯誤、權限請求）觸發時：

- 場景系統暫停當前場景（不論是探索或戰鬥）
- 將控制權交給互動事件系統
- 事件處理完成後，恢復原場景狀態

## 六、設計決策說明

### 6.1 為什麼只有兩種場景而非多種？

**問題**：複雜的多場景系統會讓用戶困惑，不知道自己當前處於什麼狀態。

**決策**：嚴格的雙場景模型 — 探索或戰鬥，非此即彼。靜止之間作為「暫停層」疊加在任意場景之上，不算獨立場景。

**效果**：用戶隨時都能清楚知道自己在「探索中」還是「戰鬥中」，可用操作一目了然。

### 6.2 為什麼探索模式和戰鬥模式的資源恢復速率不同？

**問題**：如果兩種模式恢復速率相同，用戶沒有動機在戰鬥間回城休整。

**決策**：探索模式快速恢復（HP +2/s, MP +1/s），戰鬥模式幾乎不恢復（HP 0/s, MP +0.1/s）。

**效果**：
- 鼓勵「探索→戰鬥→回城恢復→再出發」的 RPG 循環
- 戰鬥中資源有限，增加策略性（何時用技能、何時節省 MP）
- 符合傳統 RPG 直覺：城鎮休息、野外消耗

### 6.3 為什麼需要 Prompt 分析器？

**問題**：用戶不會主動告訴系統「這是一個任務」或「這只是閒聊」。

**決策**：內建 Prompt 分析器，通過關鍵字列表和複雜度評分自動判斷。

**效果**：
- 對話型 Prompt 不會意外觸發戰鬥
- 簡單任務（複雜度 < 8）由主 CLI 直接處理，不浪費戰鬥流程
- 自動化判斷，用戶無需額外操作

### 6.4 為什麼逃跑要消耗 30% MP？

**問題**：如果逃跑沒有代價，用戶遇到困難就會直接逃跑，失去挑戰意義。

**決策**：逃跑消耗當前 MP 的 30%，且不獲得任何獎勵。Worktree 戰鬥分支保留供後續處理。

**效果**：逃跑是有代價的選擇，但不至於毀滅性。保留 Worktree 讓用戶可以稍後繼續處理未完成的任務。
